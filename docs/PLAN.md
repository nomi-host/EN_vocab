## 1. 개요

- **목표**: JP vocab(ピッ!とVOCA) 앱의 검증된 구조를 복제·확장한 **개인용 영어 학습 웹앱(PWA)**.
- **핵심 축 3가지**: ① 망각곡선 기반 단어 암기 ② 퀴즈 ③ 회화력(작문 첨삭 + 페르소나 회화).
- **아키텍처**: JP 앱과 동일.
  - 단일 `index.html` (React CDN, 빌드 없음) + `words.js` 등 시드 데이터 + PWA(manifest).
  - Vercel 서버리스 프록시 `api/proxy.js` (Gemini + Google TTS 키 은닉, CORS/레이트리밋).
  - 저장: `localStorage`(학습 상태) + `IndexedDB`(TTS 오디오·단어정보 영구 캐시 → API 호출 최소화).

### 2.2 판단 근거

- **단어정보**: 뜻·발음·유의어·반의어까지는 무료 API 조합(Free Dictionary + Datamuse)으로 충분. 단, 실시간 API 의존 대신 JP 앱처럼 **시드 데이터에 미리 굽고(words.js/synonyms.js), 없는 것만 런타임 API + IndexedDB 캐시**가 안정적.
- **한국어 뜻**: 네이버/다음은 공개 API를 제공하지 않고 크롤링은 약관 위반 소지 → 대신 **오픈 사전 3종 병합(2.3 참조)**으로 대부분 커버하고, 빈 곳만 LLM 번역. 학습 중 발견되는 오역은 JP 앱의 오류 신고 흐름으로 수정.
- **어원/어근**: "같은 어원 공유 단어 타고타고"는 etymology-db에서 라틴/그리스 어근 기준으로 역인덱스를 만들어 시드화. 어근·접사 카드(com-, inter-, -tion 등)는 양이 작아 자체 데이터로 관리하는 편이 품질이 좋음.
- **문법정보**: "긁어올" 무료 소스가 없는 영역이지만 LLM도 불필요 — **브라우저 내 POS 파서(MIT 라이선스) + 규칙 매칭으로 태그를 자동 검출**하고, 태그별 설명문(관계대명사, 가정법, to부정사… ~100개)만 1회 자체 작성. 규칙이 못 잡는 예외만 LLM 폴백.
- **외부사전 연결**: URL 스킴으로 무료 연동 — 네이버 영어사전(`en.dict.naver.com/#/search?query=단어`), Longman, Cambridge, Etymonline(어원 확인용).

### 2.3 LLM 최소화 파이프라인 (생성 → 번역으로 격하)

토큰 비용과 QA 부담을 줄이기 위해 **LLM의 역할을 "콘텐츠 생성"에서 "번역·갭 필링"으로 축소**한다.

1. **표제어·레벨**: CEFR-J Wordlist(A1–B2) + Octanove Vocabulary Profile(C1–C2) — 무LLM
2. **뜻(한국어)**: 오픈 사전 3종(한국어판 위키낱말사전, 영어판 번역 섹션, 국립국어원 한영 학습사전 역인덱스) 병합 → 커버리지 측정 → **미커버 단어만 LLM 번역** (상위 5,000 단어는 오픈 사전 커버리지가 가장 높은 구간이라 갭이 작음)
3. **예문**: Tatoeba 영어 코퍼스에서 표제어별로 길이·난이도 필터로 선별(무LLM) → 영한 페어가 이미 있으면 그대로, 없으면 **LLM은 한국어 번역만**
4. **유의어·반의어·품사 파생·어원**: WordNet + etymology-db에서 기계 추출 — 무LLM
5. **문법 태그**: 브라우저 내 POS 파서 + 규칙 매칭 — 무LLM. 설명문 ~100개만 1회 작성
6. **검수(QA)**: 생성물 검수가 아니라 **번역 검수만** 남음. 번역은 원문 대조가 가능해 검수 난이도가 낮고, Tatoeba 인간 번역·사전 출처 데이터는 검수 자체가 불필요

예상 배치 비용: 예문 번역 ~1만 문장 + 뜻 갭 필링 → **1회 $2 미만**. 런타임 LLM 사용처는 학습 데이터가 아닌 기능(첨삭·회화·AI 질문)으로 한정된다.

라이선스 주의: 위키낱말사전·국립국어원 사전 파생 데이터는 CC BY-SA(동일조건) — **데이터 파일(words.js 등)에 출처 고지 주석을 넣고 해당 파일은 CC BY-SA 유지**(코드와 분리). 공공누리 1유형은 출처 표시만으로 상업 이용까지 허용.

### 2.4 다의어 원칙 — 학습 단위는 단어가 아니라 "의미(sense)"

영어는 한 표제어에 뜻이 여럿(save = 구하다/아끼다/저장하다/모으다…)이므로, **의미별로 별도 항목으로 분리해 별도 예문을 붙이고 SRS도 의미 단위로 돈다** (선행 어휘앱(국내 대표)와 동일 원칙).

- **스키마**: 표제어에 `senses[]` — 의미마다 `{cue(맥락 큐: "위험으로부터"), ko, en(영영), pos, lv(의미 빈도 레벨), ex[]}`. 단일 의미 단어는 기존 flat 필드 유지.
- **학습 단위**: 카드·SRS 박스·산출 큐 모두 `단어#의미` 단위. "학습에 추가"도 의미별 버튼.
- **출제 규칙**: 같은 단어의 다른 의미는 **같은 세션에 함께 출제하지 않음** — 유사 항목 동시 학습은 간섭(interference)으로 기억을 방해한다는 어휘습득 연구 원칙. 의미 간 간격을 두고 순차 도입(빈도 높은 의미부터).
- **파이프라인**: WordNet synset이 의미 분리를 기계적으로 제공하나 과분할이 심함 → **의미 빈도 상위 N개로 병합·선별**(단어 빈도-의미 분포 기준) 후 한국어 뜻·맥락 큐는 오픈 사전 3종 + LLM 갭필. 맥락 큐는 짧은 한국어 부사구로 통일("위험으로부터", "나중에 쓰려고").
- **표시**: 사전 행 = "구하다 외 5개 뜻" + `뜻 6` 뱃지. 상세 = 의미 카드 리스트(첫 의미 펼침, 탭 토글).

## 4. 기능 명세

### 4.1 단어 학습 (JP vocab 구조 복제 + 확장) `[부분 구현]`

JP 앱에서 그대로 복제: Leitner 망각곡선 SRS(하루 신규 N개 + 자동 복습), 단어/예문 TTS, 흘려듣기, 잔디(스트릭 캘린더), 통계&설정, 오류 신고, 업데이트 안내(WHATS_NEW), 카테고리(JLPT → **CEFR A1~C1 + 카테고리별 단어장(비즈니스/여행 등) + 내 단어**), 온보딩 레벨별 신규 학습 비율.

**수동 단어 추가 (차별점)**: 선행 어휘앱(국내 대표)류는 자사 DB에 있는 단어만 저장 가능하지만, 이 앱은 JP 앱처럼 **DB에 없는 단어도 자유롭게 추가** — 오픈 사전(시드+Free Dictionary)에서 자동 조회하고, 없으면 LLM이 뜻·예문을 즉석 생성(사용자 트리거의 소량 호출이라 2.3 LLM 최소화 원칙과 충돌 없음). 추가된 단어도 SRS·퀴즈·타고타고에 완전 편입.

**진입점 = 사전 탭(5.3)**: 검색창에 입력 → DB에 있으면 단어 상세 팝업, 없으면 "이 단어 추가하기" 버튼 노출(회화·작문 중 모르는 단어를 만나도 말풍선 단어 탭 → 같은 흐름). 사전 탭의 목록 뷰: **전체**(CEFR/주제 필터) · **학습 중** · **학습 완료** · **내가 추가한 단어** — 각 항목은 SRS 상태(박스 단계·다음 복습일) 뱃지 표시.

**단어 상세 화면 확장** (타고타고 공부의 핵심):

- 뜻(한/영), 발음기호, 품사, CEFR 레벨, 유의어/반의어(탭하면 해당 단어 상세로 이동)
- **예문에 사용된 단어**: 예문 내 단어 칩 → 탭하면 그 단어 상세로
- **연관 단어**
  - 같은 어원을 공유하는 단어 (spect → inspect, respect, spectacle…)
  - 같은 단어의 다른 품사 (decide → decision, decisive, decidedly)
- **어근/접사 칩**: `inter-` `com-` `-tion` 등 → 탭하면 그 어근을 쓰는 단어 목록 화면으로 (목록에서 또 타고 들어가기)
- 외부사전 링크(네이버/Longman/Cambridge/Etymonline), 즐겨찾기, 오류 신고

### 4.2 학습 카드 (레퍼런스 이미지 방식) `[완료]`

선행 어휘앱(국내 대표)식 "예문 빈칸 입력" 카드:

1. 상단: 상황 설명(한국어) + 한국어 문장(타깃 부분 강조)
2. 하단: 영어 예문에서 타깃 단어(구)만 빈칸 → **키보드 타이핑 또는 음성 모드로 입력**
3. 정답 시: 빈칸 채워지며 강조, 다음 복습 시점 표시("6분 뒤 복습")
4. 정답 후 액션 바: 다른 예문 / 오답 노트 / 말해보기(발음 따라하기) / 단어 정보 / AI 질문 답변
5. 하단 버튼: 느리게 듣기(x0.8) / 다시 듣기 / 다음 문제
6. 즐겨찾기(하트), 길이 보기(글자 수 힌트)

### 4.3 레벨 테스트 — 어휘력 + 문장구사력 이원 측정 `[설계만]`

> **CEFR 밴드 표기는 12단계(2026-07-13 확정)**: A1/A1+/A2/A2+/B1/B1+/B2/B2+/C1/C1+/C2/C2+. 6밴드를 base/+ 로 나눈 구성. Part A(어휘력)·Part B(문장구사력) 결과 표기에 **둘 다** 이 12단계 스케일을 쓰되, 두 점수는 계속 **완전히 분리된 두 축**으로 각자 산출·표시(하나로 합치지 않음 — "어휘는 B1+, 문장 구사는 A2" 처럼). 듣기 난이도(4.3.5의 `LISTEN_LEVELS`)에 이미 같은 12단계를 적용해뒀고, 레벨 테스트 착수 시 그 배열을 공유 상수로 재사용. words.js의 cefr 필드(8,638단어, 6단계)는 이 확장과 무관 — 레벨테스트·듣기난이도 등 **판정/표시용**으로만 12단계를 쓰고 시드 데이터 자체는 바꾸지 않음.

**Part A. 어휘** (기존 설계)
- CEFR 밴드별 적응형 체크(가짜 단어로 허수 방지) + **인지(뜻 알기) / 산출(한국어 보고 영어 말하기) 이원 측정**
- 결과: 추정 어휘량, 표준어휘 커버리지 %, 상위 %, **CEFR 12단계 밴드**(위 참조)

**Part B. 문장구사력** (어휘만으로는 "책으로 배운 사람"의 실제 회화력을 못 잡는다 — 산출 능력을 직접 측정)

1. **문장 따라 말하기 (Elicited Imitation)** — 문장을 듣고 그대로 재현(음성). 문장이 작업기억 용량을 넘으면 **내재화된 문법 없이는 재현 불가**하므로, SLA 연구에서 검증된 암묵적 문법 지식 측정법. 길이·복잡도 점증 8문항, STT 대조로 자동 채점.
2. **한→영 순간영작** — 한국어 단문을 30초 안에 영어로(음성/타이핑) 4~6문항. LLM 루브릭 채점: 어순 / 시제·상 / 관사 / 자연스러움 4축.
3. **어순 배열 스피드** — 청크를 순서대로 배열, **반응시간 기록** 5문항. SOV→SVO 재배열 자동화 수준을 계량.

**결과 통합**
- "어휘는 B1+, **문장 구사는 A2**" 식(12단계 밴드)으로 격차를 명시 — 타깃 사용자의 전형 프로필을 첫 화면에서 보여주는 것 자체가 동기부여이자 진단.
- 구사력 프로필(정확도·자동화 속도·복잡도)에 따라 **루틴 배합 자동 조정**: 구사력이 낮으면 어순 드릴·순간영작·청크 카드 비중 증가.
- 이후 학습 이력(SRS 정답률·회화 채점)으로 두 축 모두 지속 갱신 → 통계 그래프.
- 온보딩 시 1회 실시 → 신규 학습 단어 레벨 비율 결정(JP 앱 LEVEL_RATIOS 방식). 어휘 파트만 먼저 하고 구사력 파트는 건너뛰기 허용(진입 장벽 완화), 미실시 시 첫 주 학습 데이터로 추정.

**전국 분포 상위% 표시(2026-07-14 확정)**: "대한민국 영어학습자 상위 몇%" 형태로 동기부여용 분포를 보여주되, 실제 전국 표본 데이터가 없으므로 **공개 통계(토익 점수 분포 등 공신력 있는 공개 자료) 참고 추정 정규분포**를 쓰고 화면에 항상 **"추정치"** 라벨을 명시한다(실사용자 데이터가 쌓이면 자체 분포로 전환 가능). 어휘력·문장구사력 각각 독립된 곡선 + "나" 위치 마커로 표시(합산하지 않음). 와이어프레임 v4에 반영: **화면 19**(레벨 테스트 진행 — Part A/B 탭, 적응형 문항), **화면 20**(결과 — 어휘력/구사력 이원 분포 곡선). 화면 1(홈)의 백분위 표시도 이원화(어휘력 상위%·문장구사력 상위% 별도 박스). 공개 통계 소스 선정·분포 파라미터(평균/표준편차) 결정은 0.5절 모델 배정표 참고.

**엔진 구현 메모(2026-07-14, Fable 5 — index.html "레벨 테스트 엔진 (4.3)" 섹션)**:
- **다의어(senses) 배치와의 간섭 검토 결론**: 간섭 있음 — run=달리다(lv1)/운영하다(lv3)/출마하다(lv8)처럼 의미별 난이도가 다르므로, 뜻을 보여주는 문항(4지선다·산출)이 어느 의미를 제시하느냐가 채점 공정성에 직결. 대응: `ltAnchor()`가 **다의어면 항상 senses[0](최빈 의미)의 뜻으로 앵커**, 없으면 flat ko 폴백 + 밴드 풀은 **단의어 우선 샘플링** — 배치가 42개든 수천 개든 동작 동일(배치 완료 후 재작업 불필요). 단어 자체의 cefr(밴드 샘플링)·어휘량 추정은 word family 단위라 배치 무관.
- **Part A 적응형**: 1-up-1-down 계단식(A2 시작), 실단어 18문항 + 가짜 단어 3개(`LT_FAKES` 12종 큐레이션, 실단어 충돌 0 검증) + 산출 4문항. know("아세요?") → "뜻 압니다"면 같은 문항의 2단계로 뜻 4지선다 확인(자기보고 검증) — 둘 다 통과해야 정답. 능력치 = 궤적 후반 10문항 밴드 평균 ×2 → 12밴드. 어휘량 = 밴드별 정답률(허수 보정: 가짜단어 오인지율 fa로 (c−fa)/(1−fa), fa=1이면 0) × 밴드 단어 수 합산.
- **산출 문항 출제 밴드 버그 수정**: 계단식 현재 밴드에서 뽑으면 정답 직후라 대개 실력 바로 위(모르는 단어)에서 출제돼 능동 어휘가 항상 0으로 측정되는 편향 발견(시뮬레이션에서 검출) → **마지막으로 맞힌 밴드(lastOkBand)에서 출제**로 수정. 산출의 목적은 "아는 단어를 낼 수 있는가"(4.12-1 수동/능동 분리)이므로.
- **Part B v1 범위**: 어순 배열 스피드만(청크 탭 배열, 오탭 수·단어당 초 기록 → 12밴드, 속도 보정 ±0.5밴드). 따라 말하기(EI)·순간영작은 STT/LLM 프록시 미구현이라 보류 — 결과에 `partial:true`로 "부분 측정" 명시. 건너뛰기 허용(진입 장벽 완화, 기존 설계대로).
- **분포 파라미터(추정치, Fable 결정)**: 어휘량 평균 4,600/SD 2,100(국내 대학생 VST 연구들 평균 5~6천 워드패밀리에서 전체 성인 학습자로 하향 조정), 구사력 12밴드 인덱스 평균 3.0(A2+)/SD 2.2(수능형 산출<인지 프로필 반영). `LT_DIST` 상수에 근거 주석과 함께 기록, 상위%는 1~99로 클램프(0%/100%는 표본 없는 추정치로서 부정직).
- **시뮬레이션 검증(30회 평균)**: A2 학습자→A2+(어휘 2,450/능동 1,758), B1→B1+(4,470/2,907, 상위 52%), C1→C1+(7,710, 상위 7%), 전부-"알아요" 허수 응답→fa=1, 어휘량 0으로 완전 보정. Part B 3케이스(느림·오답多→A1 / 빠른 전부정답→B1+ 상위 18% / B1 문장만 성공→A2+ 50%) 합리적 판정 확인.
- 결과는 `localStorage["en_leveltest_v1"]`에 저장(`saveLevelTest`/`loadLevelTest`) — 홈 더미 데이터 실데이터 전환 작업(Fable 5 예정) 때 홈 3칸(CEFR·어휘량·상위%)이 이걸 읽도록 연결.

**2차 개정(2026-07-14, 사용자 피드백 7건)**:
- **뜻 4지선다 오답을 같은 품사로 통일**(`ltPos` — 품사가 섞이면 소거법으로 너무 쉬움. 같은 품사 풀 8개 미만이면 밴드 전체 폴백). 시뮬 검증 오답 품사 일치율 100%(1,071/1,071).
- **know↔meaning 전환 시 카드 크기·폰트 고정**(`.lt-qcard` min-height 150 + `.lt-word` 24px 동일 타이포 — 화면 흔들림 제거).
- **한 단계 뒤로가기**(엔진 `undo()` — 매 응답 직전 전체 상태 스냅샷을 쌓고 복원. "실수로 안다고 눌렀을 때" 같은 문항으로 되돌아감). 진행바 옆 chevron 버튼, 첫 문항에선 비활성.
- **산출(타이핑) 문항에 모름 버튼** — 빈 답 제출로 오답 처리.
- **Part B도 적응형으로 전환**: 고정 [A2,B1,B1,B2,B2] 순서 폐기 → A2 시작, 오탭 1개 이하 완성이면 한 단계 위 문장/아니면 아래(엔진이 문항을 동적으로 뽑음, 셔플이 정답 순서와 같으면 재셔플). 시뮬: 완벽 사용자 A2→B1→B2→C1→C2 상승 확인.
- **Part B 칩 자리 유지**: 탭한 칩이 사라지며 남은 칩이 당겨지던 것을 → 자리 유지한 채 흐려지기만(`.picked`, 오탭 방지) + **실행취소 버튼**(마지막 선택 해제).

### 4.3.5 듣기 세션 (딕테이션 — 루틴 2단계) `[부분 구현 — 2026-07-13]`

선행 어휘앱(국내 대표)의 듣기 훈련을 독립 세션으로 채택. **문장을 듣고 그대로 타이핑**하는 것이 기본 흐름.

1. 오디오 재생(문장 전체) → 타이핑 입력(전체 문장 한 번에) → 채점(구두점 무시·단어 단위 정오 대조) → 정답 문장 표시 + 해석. (단어 수만큼 빈칸으로 미리 보여주는 프리뷰는 미구현)
2. 보조 컨트롤: **다시 듣기 · 느리게 듣기(0.7x) 완료**. 또박또박/자연 속도(연음) 토글, 못 잡은 단어만 부분 재생은 미구현
3. 출제 소스: **학습한 단어(SRS 기록 있음)의 예문 완료**. 즐겨찾기 문장·회화 수정문 연동, 듣기 정답 시 SRS 복습 가점은 미구현
4. 세션당 10문장(퀴즈와 동일 진입점, 학습 탭 메뉴에서 시작). 루틴 자동 편입, 문장 길이·속도 기반 난이도 조절은 미구현
5. 오답 노트 축적(퀴즈·흘려듣기 공유)은 미구현 — 정오답이 SRS에는 영향 없는 순수 보조 훈련으로 우선 구현

**구현 메모(2026-07-13, 3차 개정 — "퀴즈 아니라 학습 형식"으로 흐름 자체를 변경)**:
- **완전정답 게이트**: 딕테이션은 학습카드처럼 "학습 형식" — 한 단어라도 틀리면 다음 문장으로 못 넘어가고 같은 문장을 다시 시도해야 함(퀴즈처럼 오답이어도 그냥 다음으로 넘어가는 방식 아님). 오답 시 정답은 공개하지 않고 **어느 칸이 틀렸는지만** 칸 색으로 표시(빨강=오답/초록=정답, 입력값은 유지)하고 재시도 유도. 완전히 맞혀야 정답 화면(해석·해설·다음 문장 버튼)이 열림. SRS는 첫 시도에 맞혔을 때만 "정답"으로 채점(재시도 끝에 맞힌 건 "오답"으로 채점 — 다음 복습을 더 앞당김).
- 빈칸: 정답 글자 수만큼 폭이 고정된 통짜 블록(입력해도 안 늘어남, 글자 단위로 나누지 않음) — 유지.
- **음절보기 힌트 재설계**: 1차로 만들었던 "학습카드 글자수 힌트(자모 하나씩 나뉜 셀)" 재사용은 오독이었음 — 음절은 글자 수가 아니라 **소리 단위**라 하나씩 끊으면 안 됨. 모음 그룹 수로 음절 수를 대략 추정하고, 단어 아래 그 개수만큼 **얇은 바(막대) 몇 개**로 "대략적인 길이감"만 보여주는 것으로 교체(`syllableChunks`) — 입력창 자체는 그대로 통짜 블록 1개, 막대는 순전히 시각 보조.
- 정답 확인 후: 문장 속 단어를 탭하면 그 단어의 사전 상세 모달이 열림(사전에 없는 단어는 비활성). 오디오 재생은 항상 문장 단위(다시듣기 버튼)만.
- **발음 해설 재설계**: "gimme/gonna" 같은 이미 잘 알려진 축약형 사전 목록은 제거(너무 뻔함) — 대신 그 문장 안에서 실제로 **자음으로 끝나는 단어 바로 뒤에 모음으로 시작하는 단어가 오는 지점**(연음/리에종)을 찾아 그 두 단어 자체로 해설을 만듦(`findLiaisonNotes`, 예: "with others"). 문장마다 다른, 그 문장 고유의 연음 지점을 짚어주는 방식 — 특정 단어쌍의 실제 블렌딩 발음(예: "위더스")까지 정확히 표기하려면 IPA 기반 음성 지식이 필요해 과장·오류 위험이 있어 이번엔 "여기서 이어져 들려요" 수준의 일반 규칙 설명으로 한정(LLM/서버 없이 클라이언트 완결).
- 1번은 동작(학습카드와 동일한 `pickSession` 재사용 — due→fresh 우선순위 풀 공유). 3번(출제 소스)은 "학습한 단어의 예문"만 반영 — 즐겨찾기·회화 수정문은 그 기능 자체가 아직 없어 제외. 5번(오답 노트)은 퀴즈가 없어 보류 — 퀴즈(4.4) 착수 시 함께 설계.

**구현 메모(2026-07-13, 4차 개정 — 힌트 위치 통일 + 듣기 난이도를 어휘 CEFR과 분리)**:
- 힌트(다시듣기/음절보기) 버튼을 학습카드(CardSession)와 같은 위치(카드 아래, 확인 버튼 바로 위)로 이동 — 기존엔 카드 맨 위에 있어 두 세션의 레이아웃이 어긋나 있었음.
- **문제마다 "듣기 난이도" 뱃지 추가** — "r u all good?"류처럼 단어는 전부 쉬운데(어휘 CEFR 기준 A1) 연음·문장 길이 때문에 실제로 듣기 어려운 문장을 어휘 레벨만으로는 못 잡는다는 지적 반영. `sentenceCefrLevel`(어휘)과는 별개로 `listeningDifficulty`를 신설: 어휘 레벨(0~5) + 연음 지점 수×1.2 + 8단어 초과분×0.3을 합산해 판정, 카드 상단에 뱃지로 표시.
- **듣기 난이도 스케일을 12단계로 확장(2026-07-13 추가 반영)** — 레벨테스트(4.3)에서 쓰기로 한 "A1/A1+/A2/A2+…C2/C2+" 12단계와 맞춤. `LISTEN_LEVELS`(12개) 도입, 위 합산 점수를 0.75 구간폭으로 12단계에 매핑(`Math.round(score/0.75)`, 경험적 근사치). **words.js의 cefr 필드(8,638단어)는 그대로 6단계 유지 — 이번 확장은 듣기 난이도 판정·표시 전용**(사용자 확인: "지금은 듣기 난이도만 12단계로"). 뱃지 텍스트는 12단계 그대로 노출하되, 색상은 가독성을 위해 4개씩 묶어 3톤(초록/주황/빨강)만 재사용. 실측: "I saw a dog in the park."→A2+, "Children learn to interact with others."→B1, "Are you all okay in an old apartment?"(전부 A1 어휘, 연음 4곳)→C1+, "The comprehensive analysis of an intricate economic infrastructure…"(고난도 어휘+긴 문장)→C2+ — 어휘 난이도와 무관하게 연음·길이가 반영됨을 확인.
- **다시듣기 패널티도 `listeningDifficulty` 기준으로 전환**(이전엔 어휘 CEFR인 `sentenceCefrLevel` 기준이었음) — 다음 문제를 고를 때도 "듣기가 더 쉬운" 문제로 교체하는 것이 목적에 더 맞음. 12단계 확장으로 다운그레이드도 더 세밀한 단위로 동작.
- `liaisonPoints`(연음 지점 전체 탐색)를 공용 함수로 분리해 해설 표시(2개 캡)와 난이도 계산(전체 개수) 양쪽에서 재사용.

### 4.4 퀴즈 (보조 훈련 — 루틴 외 선택) `[단어 퀴즈 완료 · 문장 퀴즈 미착수]`

- **단어 퀴즈**(완료, 2026-07-13): 학습한 단어(SRS 기록 있는 것) 대상 ① 뜻 맞추기(4지선다, 오답 3개는 전체 표제어 풀에서 무작위) ② 스펠링 맞추기(발음 듣고 타이핑) — 절반씩 섞어 10문제. 대상 단어 4개 미만이면 메뉴 비활성. 정오답이 SRS 복습 간격에는 영향 없음(순수 보조 훈련).
- **문장 퀴즈**: 학습한 예문/즐겨찾기 문장 대상. 제안 방식(효과 순):
  1. **빈칸 채우기(cloze)** — 기본. 타깃 단어+콜로케이션 단위로 구멍
  2. **딕테이션** — 듣고 받아쓰기(전체 또는 부분), 청취력 동시 훈련
  3. **어순 배열** — 문장을 청크로 쪼개 순서 맞추기(어순 감각)
  4. **한→영 순간작문** — 한국어 문장 보고 음성으로 말하기, LLM 채점 (회화 연계, 상급)
  - 기본은 1+2 혼합 출제, 3·4는 설정에서 켜는 방식 제안
- 하루 10문제 이상 → 잔디 반영(JP 앱 규칙 복제)

### 4.5 작문 첨삭 (회화력 축 1) `[설계만]`

**설계 변경(2026-07-12): 회화(4.6)의 메신저형 문답 왕복과 겹치지 않도록, 작문 첨삭은 발화가 아니라 "쓰기" 장르로 범위를 고정한다.**
회화는 실시간 유창성(짧은 발화 주고받기)을 훈련하고, 작문은 격식·구조·논리 전개(긴 글 조립) 를 훈련 — 서로 다른 생산 스킬이라 UI도 분리.

- **장르 선택**(카테고리 대체): 이메일, 업무 메신저 답장(슬랙/팀즈 톤), 일기, SNS 포스트, 자기소개서 등 — 실제로 쓸 일이 있는 문서 단위. 여행지 스몰토크·공항 같은 발화 상황 카테고리는 4.6 회화 페르소나 쪽으로 이관.
- **UI**: 채팅 말풍선이 아니라 **본문 에디터 + 여백 첨삭**(구글독스 코멘트/Grammarly식) — 문장이 아니라 문단·문서 단위로 작성하고, 문장별 인라인 코멘트가 여백에 달림.
- 흐름: 장르+상황 프롬프트 제시(예: "동료에게 마감 연장을 요청하는 이메일") → 사용자가 영어로 초안 작성(타이핑) → LLM이 문서 전체를 첨삭
- 첨삭 결과: **자연스러움 %**, 문장별 교정(diff 강조 인라인 코멘트), **격식 지수**(너무 캐주얼/너무 딱딱함 여부 — 장르별 기준 다름), 더 자연스러운 대안 표현, 전체 총평 한 줄
- **문장 즐겨찾기** → 단어장(문장 탭)에 저장
- 내 작문 이력/장르별 평균 자연스러움 통계
- 회화 탭(4.6)에서는 "작문 연습" 진입점만 유지하되, 탭하면 채팅방이 아니라 이 에디터 화면으로 이동(발음 클리닉만 대화형 유지 — 말하기 훈련이라 채팅 형식이 적합).

### 4.7 단어장 (즐겨찾기 허브) `[부분 구현]`

**2026-07-12 구현 메모**: 즐겨찾기·내 단어장 실제 데이터/조작은 **사전 탭 안의 "내 단어장" 메뉴**로 구현됨(계속 그 위치 유지 — 검색·카테고리 목록에서 바로 담을 수 있어야 해서 사전 안이 자연스러움).
전체 표제어 중 원하는 단어만 골라 **이름 붙인 커스텀 리스트**(예: "비즈니스용")로 저장 — 사전 목록(카테고리별·즐겨찾기·검색결과 등)에 다중선택("담기") 모드로 여러 단어를 한 번에 담거나, 단어 상세 모달에서 개별로 담을 수 있음. 리스트 상세에서 "이 리스트로 학습" → 그 리스트 단어만으로 단어카드 세션 시작. 리스트 이름 변경·삭제·단어 제거 가능. 사전에 없는 단어를 검색해서 채워 넣는 기능(PCM류 앱 참고)은 미착수 — 사전 API 연동 필요.

**2026-07-13 구현 메모**: 보관함 탭에 **뱃지 컬렉션 그리드**(BADGES 데이터를 4열 그리드로 전부 노출, 탭하면 홈과 동일한 상세 다이얼로그) + **즐겨찾기/내 단어장 바로가기**(탭하면 사전 탭으로 이동) 추가. 뱃지 자체의 획득 조건·종류 확정(12종 체계)은 별도 설계 작업(Fable 5 담당, 태스크 #19) — 지금은 기존 데모 BADGES 7개를 그대로 그리드로 보여주는 표시 레이어만 구현.

- 탭 2개: **단어** / **문장**
- 단어 탭: 즐겨찾기 단어 → 단어 상세로 타고타고 진입
- 문장 탭: 작문·회화·카드에서 저장한 문장. 상세에서:
  - 문장 TTS, 번역, 출처(회화: 어느 페르소나와의 대화인지)
  - **사용된 단어 칩**(→ 단어 상세), **사용된 문법 태그**(→ 문법 설명 + 같은 문법의 다른 저장 문장)

### 4.8 발음 진단 `[미착수]`

- 흐름: 작문 첨삭·회화에서 **수정문을 받으면 → 따라 말하기 → 정확도 진단**. 학습 카드의 "말해보기"도 같은 엔진 사용.
- 엔진: **Azure Speech Pronunciation Assessment** — 음소/단어/문장 단위 정확도(accuracy) + **프로소디(강세·억양·속도·리듬)** + 유창성 점수를 반환. 발음 평가의 사실상 업계 표준(언어학습 앱 다수 사용).
  - 비용: 표준 STT 요금과 동일 수준(시간당 약 $1) → 하루 10분 진단 시 **월 $5 미만**. 개인용 무리 없음.
- 화면: 문장 위에 **단어별 색상(정확/부정확)** + 틀린 음소 표시 + 강세 위치 비교(모범 vs 내 발음). 낮은 단어는 탭하면 모범 발음 재생 + 재시도.
- 자체 로직 검토 결과: Web Audio로 피치·음절 길이를 추출해 **강세 패턴 비교(시각화)는 클라이언트에서 가능** — 보조 그래픽으로 채택. 그러나 음소 단위 정확도는 자체 구축이 비현실적(강제정렬+음향모델 필요) → Azure 채택이 결론.
- 모범 음성 TTS 업그레이드 옵션: 기본은 Google Neural2 유지. 발음 진단 모범 음성만 고품질 엔진 검토 — Azure HD($30/1M자)·OpenAI($15/1M자)·ElevenLabs(최고 품질, 고가). 진단 문장은 짧고 캐시되므로 어느 쪽이든 비용 미미. **1차는 Neural2로 충분, 품질 불만 시 교체.**

**발음·표현 진단 확장 (2026-07-13 확정, 위 Azure 엔진 위에 얹는 기능)**
- **진단 카테고리(한국어 라벨)**: Azure가 주는 점수 축을 학습자용 카테고리로 노출 — **정확도**(음소·단어 단위), **유창성**(끊김·속도), **억양·리듬**(프로소디). 여기에 LLM 기반 **자연스러움(idiomatic)** 축 추가 — 전사된 문장의 어휘·표현 선택을 작문 첨삭과 같은 방식으로 평가(발음 엔진이 아니라 LLM 담당).
- **발음 레벨화 + 홈 노출**: 위 축들을 종합한 발음 레벨(레벨 테스트 12단계 체계와 동일 척도)을 산출해 홈 레벨 카드(현재 CEFR/어휘량/백분위)에 발음 축 추가. **회화·발음 진단을 진행할수록 점수 이력이 누적돼 레벨이 갱신**됨(통계 그래프에도 추이 표시).
- **정성적 억양 코칭**: 한국어 화자에게 흔한 패턴(R/L 구분, 종성 첨가, th 발음 등)이 감지되면 코칭 코멘트로 안내. ※ "어느 나라 억양 몇 %" 식의 정량 진단은 상용 API에 존재하지 않고 자체 구축도 비현실적이라 **채택 안 함**(보류) — 정성 코멘트가 현실적 대안.

### 4.10 동기부여 시스템 (저공수 우선) `[부분 구현]`

채택 (디자인·개발 공수 최소 순):

1. **캐릭터 성장 스테이지** — 누적 학습·스트릭에 따라 홈 캐릭터 이미지가 5단계로 변화. 유령의 "NO ENGLISH" 팻말 문구가 점점 바뀌는 서사(예: NO ENGLISH → OH... ENGLISH? → OK ENGLISH). 개발 = 이미지 스왑 로직뿐, 디자인 = 5컷. **공수 대비 효과 최고.**
2. **뱃지 컬렉션**(구 "스카우트 조끼판" — 2026-07-12 명칭 변경, 조끼 은유 제거) — 조건 달성 시 뱃지 획득, 보관함의 "뱃지 컬렉션" 화면에 표시. 개발 = 조건 체크 + 컬렉션 화면. 뱃지 아이콘은 사용자(디자이너) 강점 활용.
   - **UI 스펙 확정·구현(2026-07-15) — 애플워치 운동 + 피크민 블룸 참고, 원형 랭크 메달 시스템**:
     - **카테고리 × 랭크 티어 구조**: 그리드(구 12칸) 대신 **카테고리마다 5단 랭크 티어**(브론즈<실버<골드<플래티넘<마스터[레인보우]). 누적 지표 임계값을 넘으면 랭크가 오른다(피크민의 3/40/400/1000/2000처럼). 뱃지 자체는 **원형 메달**(티어별 금속 톤 링 + 이너 페이스, 중앙에 임계값 표기).
     - **현재 9개 카테고리**(8~12 목표 범위): 연속 학습 · 단어 학습 · 듣기 세션 · 회화 · 레벨 업 · 어원 탐험 · 복습 마스터 · 친구 초대 · 자립 회화(SOS 없이). 코드: `BADGE_CATS`(카테고리+임계값+설명), `BADGE_TIERS`(5티어), `BADGE_PROGRESS`(현재 더미 진행값·획득일).
     - **컬렉션 화면(보관함)**: **한 줄당 한 카테고리**(헤더 + `n/5` 획득 카운트) + **가로 스와이프로 5티어 랭크 열람**. 미획득 티어는 **실루엣(그림자 처리)** — 임계값만 흐리게 보임.
     - **홈 스테이지 트랙**: 캐릭터 뒤 가로 트랙에 **카테고리당 대표 메달 1개**(최고 획득 티어) 노출(CLAUDE.md 작업 원칙 — 홈 뱃지 트랙 유지).
     - **상세(탭 시)**: 메달이 크게 확대되고 **밀어서 뒤집기(3D flip, `transform: rotateY`)** — 앞면=메달+티어명, 뒷면=획득일(또는 조건)+설명+랭크 스트립(탭해 다른 티어 열람). **빠르게 밀면 관성으로 한 바퀴 더 빙그르르**(포인터 속도 임계값). 컴포넌트: `BadgeMedal`, `BadgeDetail`.
   - **뱃지별 획득 로직(트리거 조건)은 아직 더미 — 실제 지표 연결 미확정(세션 태스크 #19).** 현재는 `BADGE_PROGRESS` 하드코딩. 실제 스트릭·SRS·회화 로그·친구 수 등에 연결하고 **진행값을 서버 DB에 저장(기기 간 동기화)**하는 것은 6.6 백엔드 설계 범위. 특히 **친구 초대·자립 회화 등 소셜/서버 의존 카테고리는 백엔드 없이는 더미로만 존재**한다.
3. **CEFR 승급 세리머니** — 레벨 테스트 연동, 승급 시 전용 화면 1장(시골쥐→서울쥐 연출). 개발 = 조건 + 화면 1개.

보류(공수 큼): 집 꾸미기/이사, 아이템 착용 꾸미기 — 에셋 조합 폭발로 디자인 공수가 계속 늘어나는 구조라 제외. 선행 어휘앱(국내 대표)식 "꾸미기"의 동기부여는 1~2번 조합으로 대체.

### 4.11 흘려듣기 확장 — 내 대화 복습 재생 `[설계만]`

기능: 지금까지의 회화·작문에서 **AI가 수정해준 문장들을 한국어→영어 순으로 연속 재생** (기존 흘려듣기의 재생 대상에 "내 대화" 소스 추가).

**학습 효과 검증 결론: 조건부로 유효.**

- 근거 ①: 자기가 말하려다 실패한 문장은 일반 예문보다 기억 우위(자기참조 효과 + 산출 시도 후 정답 노출 = 인출 실패 후 피드백 학습).
- 근거 ②: "내 발화 → 교정 확인" 사이클의 반복 노출은 출력 가설(Swain)·주목 가설(Schmidt)이 말하는 "갭 인지"를 강화 — 교정을 받은 직후 1회로 끝나는 게 보통인데, 간격을 두고 재노출하면 SRS 원리가 문장 단위에 적용됨.
- **유효 조건 3개** (설계 반영):
  1. **수정본(정문)만 재생** — 내 오류 문장을 다시 들려주지 않는다(오류 재각인 방지).
  2. **기본 순서는 한→영** — 한국어를 먼저 듣고 영어를 속으로 인출한 뒤 정답을 듣는 구조(인출 연습). 영→한 순서는 섀도잉·복습용 옵션.
  3. 문장 사이 **2~3초 인출 간격**(무음) 삽입.
- 한계도 명시: 순수 수동 듣기 단독의 효과는 근거가 약함 → 이 기능은 "이동 시간에 하는 보조 복습"으로 포지셔닝하고, 잔디 반영은 안 함(수동 소비를 학습 실적으로 계산하지 않음).

**추가 결정(2026-07-12) — "대화 다시듣기" 모드 신설.** 저장 문장을 낱개로 흘려듣기만 하면 맥락이 끊겨 기억 효과가 떨어진다는 문제 제기 → 문장 하나만 따로 재생하기보다, **대화 단위 전체 재생**을 기본 모드로 추가한다(개별 문장 흘려듣기는 세밀 복습용으로 병행 유지).

- **진입점**: 단어장(4.7) 문장 탭 또는 회화 대화 종료 요약 화면에서 "대화 다시듣기" 버튼. 목록은 **"yy.mm.dd {페르소나}과의 대화 다시듣기"** 형식으로 날짜+상대 이름 노출 — 에피소드 기억 훅(그날 그 대화라는 맥락) 역할.
- **재생 순서**: ① 하이라이트 구간 — 즐겨찾기·교정된 핵심 문장만 골라 **영→한→영→한** 순으로 먼저 프라이밍 재생 ② 이어서 **전체 대화를 영/한 대사 그대로 2회 반복** 재생(1회차 자연 청취, 2회차 인출 확인).
- 문장 단위 흘려듣기(기존 4.11 본문)와는 별도 모드로 공존 — 문장 단위는 SRS성 낱개 복습, 대화 다시듣기는 맥락 중심 몰입 복습으로 용도 구분.

### 4.12 타깃 특화 기능 — "수능은 했는데 회화가 안 되는 사람" `[미착수]`

타깃 정의: **기초 문법·독해·듣기는 되는데(수능 학습 경험) 말하기 산출 경험이 0인 학습자.** 문제는 지식 부족이 아니라 ① 수동 지식이 입으로 안 나오고 ② 직역·문어체(수능체)로 말하며 ③ 자연 속도 연음을 못 알아듣고 ④ 막히면 침묵하는 것. 이에 맞춘 특화 기능:

1. **아는 단어 → 쓰는 단어 전환 훈련**: 레벨 테스트를 "인지(뜻 알기)"와 "산출(한국어 보고 말하기)"로 이원 측정 → **"알지만 못 쓰는 단어" 큐**를 별도로 만들어 산출형(한→영 스피킹) 카드로 우선 복습. 통계에 수동/능동 어휘량을 따로 표시.
2. **문법 자동화 드릴**: 문법을 새로 가르치지 않고(이미 앎) **아는 문법을 입으로 자동화** — 패턴별 순간영작 스피킹(가정법, 관계사, 분사구문 등 수능 문법 용어로 분류해 친숙하게). 문법 태그·설명도 수능 용어 체계로 표기.
3. **수능체 감지기**: 작문·회화 첨삭에서 문법 오류와 별도로 **"문어체/직역 티"를 하이라이트** — 격식 지수 표시 + 원어민 구어 버전 제시 (예: "However, I think that..." → "But honestly, ..."). 콩글리시 직역 패턴 사전 내장.
4. **연음·축약 듣기 훈련**: 딕테이션에 또박또박/자연 속도 토글, gonna·wanna·연결발음 구간 하이라이트. 수능 듣기 속도와 실제 회화 속도의 간극을 명시적으로 훈련.
5. **회화 SOS 버튼**: 대화 중 막히면 침묵 대신 SOS → 하고 싶은 말을 한국어로 말하면 영어 표현을 알려주고 **다시 말해보게 한 뒤** 대화 재개. "막히는 경험"을 학습 이벤트로 전환.
6. **필러·리액션 팩**: Well / Let me think / That makes sense 같은 대화 윤활유 표현을 초기 학습 큐에 배치하고, 회화 피드백에서 침묵 구간에 필러 사용을 코칭.
7. **회화 난이도 커브**: 목표가 정해진 롤플레이(입국심사 통과하기 등 스크립트형) → 반자유 대화 → 자유 대화 순. 응답 시간도 무제한 → 제한 모드로 점진 강화.

### 4.13 유창성 확장 — 언어학 검토 채택분 (근거: docs/LINGUISTICS.md) `[설계만]`

언어학 검토 결론: 기존 설계는 산출 중심이라 뼈대는 적합하나 **"정확하게"에 편중, "빠르게(자동화)·회복력 있게(협상)"가 부족.** Nation의 Four Strands 중 유창성 개발 가닥을 아래 7건으로 보강한다.

1. **청크(정형표현) SRS — 최우선**: "be alert to", "I was wondering if" 같은 콜로케이션·문두 프레임을 단어와 동급의 카드 단위로. 유창성은 단어 조립이 아니라 통짜 표현 인출에서 나온다(Pawley & Syder). 학습 카드 빈칸이 이미 구 단위라 데이터 확장으로 구현.
2. **섀도잉 모드(듣기 세션 내)**: 문장 듣고 즉시 겹쳐 말하기 5~10문장 연속 세트 — 선언지→절차지 자동화(DeKeyser). 발음 진단 파이프라인 재사용.
3. **다시 말하기 드릴(4/3/2 경량판)**: 회화 종료 후 내 파트를 수정본 기준 60초→40초 재발화. 대화 로그 재활용이라 콘텐츠 비용 0.
4. **어순 스피드 드릴 승격**: 청크 배열 + 반응시간 기록. SOV→SVO 자동화를 계량(레벨테스트 Part B-3과 데이터 공유).
5. **의미협상 페르소나**: 페르소나가 확률적으로 되묻기("Sorry, what do you mean?") — 실전 회복력 훈련(Long). 불친절 페르소나는 되묻기 빈도 상향.
6. **최소대립쌍 모듈(발음 클리닉)**: r/l·f/p·v/b 듣고 고르기 + 말해서 통과. Azure 음소 점수로 채점.
7. **회화 피드백 2모드**: 4.6에 반영(정확성/유창성 모드).

| 영포자 | 국내 타깃에 직관 최강, 검색 유리 | 한국어라 확장성 없음 — 서브타이틀로 활용 권장 |

- 확정 전 체크: 앱스토어 동명 앱, 상표검색(KIPRIS), 도메인/SNS 핸들.

### 5.2 캐릭터 시스템 (메인 + 서브, 선행 게임화 학습앱 구조) `[부분 구현]`

- **메인 후보: 유령** — "NO ENGLISH" 팻말을 든 유령. 네이밍 1안과 세계관 일치. 학습이 쌓일수록 팻말 문구·유령의 표정/형체가 변하는 성장 서사(5.4 동기부여와 연결).
- **서브 캐릭터**: 두더지, 상자 속 길고양이, 나무늘보, 시골쥐→서울쥐 — 페르소나 아바타·뱃지·빈 상태 일러스트·카테고리 마스코트로 배치. 시골쥐→서울쥐는 그 자체가 성장 서사라 레벨 승급 연출에 적합.
- **그래픽 스타일**: 원색 + 라인(아웃라인) 벡터 일러스트. 제작: 사용자.
- 사용자 추가 페르소나(4.6)의 프로필 이미지는 이 기본 캐릭터 세트에서 선택.

#### 5.2.4 캐릭터 선택 & 변경 — 포인트 잠금

- **처음 1회는 무료 선택.** 홈에 캐릭터가 없으면 "캐릭터를 선택해 주세요" 카드가 뜨고, 설정에서 고른다.
- **이후 변경은 포인트 소모형**(300P, `CHAR_CHANGE_COST`) — 자유 선택 UI를 홈에서 제거하고 **설정(기어 아이콘) 화면으로 이동**, "일정 학습량 유지"를 계량하기 위해 단어카드 정답 시에만 10P 적립(`POINTS` 엔진, localStorage). 포인트 부족 시 "0/300P"로 비활성 표시.
- 근거: 홈은 매일 보는 화면이라 즉시 바꿀 수 있는 피커가 있으면 습관적 이탈 지점이 되지만, 설정 안에 포인트 게이트로 넣으면 "캐릭터 교체"가 꾸준한 학습에 대한 보상 이벤트로 전환된다.

#### 5.2.5 캐릭터 아트 에셋 파이프라인

- 쥐 캐릭터 러닝 모션은 `img/rat_run.js`(4프레임 SVG, svgo로 52.7% 최적화)로 자산화하고 `SpriteChar` 컴포넌트가 프레임을 순환 재생(연속일 단계에 비례한 fps). 나머지 캐릭터(유령·두더지·외계인)는 실제 아트 나오기 전까지 텍스트 placeholder 유지.
- 새 캐릭터 아트 추가 시: 4프레임 SVG를 같은 방식(`window.CHAR_<NAME>_RUN` 전역 + svgo 최적화)으로 `img/`에 넣고 `SpriteChar` 분기만 추가하면 됨 — 아트 교체가 코드 구조 변경 없이 가능.

#### 5.2.1 캐릭터 로스터 & "달리는 홈" 모션

홈 첫 화면에서 **캐릭터가 계속 달려가는 모션**(제자리 달리기 + 배경 스크롤 = 전진 착각)이 핵심 정서. 선택 캐릭터 4종, 각자 달리는 방식이 다름:

| 캐릭터 | 달리기 표현 |
|---|---|
| 고양이 유령 | 스케이트보드 |
| 두더지 | 땅 파며 질주 |
| 쥐 | 짐보따리 메고 달리기 |
| 외계인 | UFO 타고 |

- **연속 학습일 → 성장 4단계**: 1단계(<3일) · 2단계(<7일) · 3단계(<14일) · 4단계(14일+). 단계가 오를수록 **달리는 속도(모션 빠르기)도 상승**.
- 구현 슬롯: `index.html`에 달리는 씬 뼈대 완성(도로 스크롤·속도선·까딱임 placeholder, 속도는 단계별 CSS 변수). **실제 캐릭터 아트만 끼우면 됨.**

#### 5.2.2 애니메이션 기술 선택 (로띠 필수 아님)

- "다리가 실제로 움직이는 달리기"는 **단일 이미지 CSS로는 불가** → 프레임 기반 필요.
- **1순위 권장: Rive** — 상태머신 하나로 "성장 4단계 + 달리기 속도 + 캐릭터별"을 파라미터(연속일)로 전환. 파일·로직 관리 최적.
- **2순위: Lottie** — 단계별 4개 JSON(After Effects+Bodymovin). 라이브러리 lottie-web(~250KB).
- **무라이브러리 대안**: 스프라이트 시트(steps() 애니) 또는 APNG/애니 WebP. 의존성 0.
- 결론: **로띠는 선택지 중 하나일 뿐 필수 아님.** 4캐릭터×4단계×달리기의 상태 관리를 생각하면 Rive가 가장 깔끔. 어느 쪽이든 현재 만든 씬 슬롯에 그대로 삽입.

#### 5.2.3 위젯 — 교통표지판 (연속일 상태 머신)

홈 스크린 위젯(네이티브, Capacitor 전환 시 WidgetKit). 연속 학습일을 **교통표지판 은유**로:

| 상태 | 표지 | 조건 |
|---|---|---|
| 위험 | **경고 삼각형(!)** | 연속 0일 / 오늘 미학습으로 스트릭 끊길 위기 |
| 정상 주행 | **속도제한 원형(숫자=연속일)** | 학습 유지 중 — 숫자가 커질수록 "빨리 달리는 중" |
| 상위 단계 | 속도제한 숫자 상승 + 캐릭터 달리기 빨라짐 | 성장 단계와 연동 |

- 앱 내 홈 스테이지 좌상단에 **동일 표지를 프리뷰**로 표시(위젯 미출시 단계에서도 콘셉트 노출, 에셋 재사용).
- "속도"와 "달리기"가 한 은유로 묶임 — 캐릭터 속도·위젯 속도제한·성장 단계가 모두 연속일 하나로 구동.

### 5.3 UX 구조 — 하단 탭 5개, 홈 중앙 배치 `[부분 구현]`

**학습 루틴 정의: 단어카드 / 듣기 / 회화(챗봇+작문)** — 홈은 이 3단계 루틴의 허브.

| 탭 (좌→우) | 내용 |
|---|---|
| **학습** | 단어카드 세션 · **듣기 세션(딕테이션)** · 퀴즈 · 흘려듣기 · 레벨 테스트 |
| **사전** | **진입 시 허브(메뉴)** — 목록을 쏟아내지 않음: ① 새 단어 등록 ② 내가 학습한 단어 ③ 카테고리별 보기 ④ 즐겨찾기 ⑤ 내 단어장. 검색은 상시 최상단(입력 즉시 결과 전환). **카테고리 솔팅 결론: 1차 레벨(CEFR — 학습 우선순위와 직결), 2차 어근별(타고타고 강점), 3차 주제별(비즈니스/여행 — 시드에 topic 필드 추가 후)** |
| **홈 (중앙)** | 캐릭터 + 스트릭 + **내 레벨 상시 표시**(CEFR·추정 어휘량·상위%) + 오늘의 루틴(단어 → 듣기 → 회화) |
| **회화** | **메신저형 대화방 목록** — 페르소나 방 + 고정 방(작문 연습·발음 클리닉) + 페르소나 추가 |
| **보관함** | 즐겨찾기 단어장(단어/문장) · 뱃지 컬렉션(카테고리별 랭크 티어 행, 탭→확대·밀어서 뒤집기 — 4.10) · 성장 기록 |

- 통계&설정은 홈 우상단 프로필 진입 유지.
- **5번째 탭 검토 결과 — "사전" 채택.** 근거: ① "없는 단어 추가를 어디서 하나"와 "학습한/전체 단어 목록 보기"가 한 화면의 같은 행위(단어 탐색)라 탭 하나로 해결 ② 사전(전체 DB·탐색)과 보관함(내가 고른 것)의 역할이 겹치지 않음 ③ 타고타고(어원·어근 네트워크)의 자연스러운 입구. 탈락 후보: 통계(프로필 진입으로 충분, 매일 볼 화면 아님), 발음(회화 탭 내 도구가 자연스러움), 랭킹(1인 앱에 무의미).

**단어 상세 네비게이션 (JP 앱과 동일)**: 목록/카드에서 단어 탭 → **팝업(바텀시트)** → 팝업 안에서 유의어·어원·예문 단어 등 타고 들어가면 **모달 스택**으로 쌓이고 뒤로가기로 한 장씩 복귀. 어디서든 동일 패턴(학습 카드·사전·보관함·회화 말풍선).

### 5.3.1 회화 탭 — 메신저 UI `[미착수]`

- 탭을 열면 **대화방 목록**(카카오톡형): 페르소나별 방 — 아바타 + 이름 + 마지막 메시지 미리보기 + 이어하기 표시.

## 6.5 개발 전략 — 처음부터 네이티브 vs PWA 후 전환

전제: 개발은 Claude(원격), 코드 자산은 JP 앱(웹), 네이티브로 가더라도 현실적 선택지는 Capacitor(웹 코드 래핑)이므로 **두 경로의 코드는 사실상 동일하다. 진짜 질문은 "언제 래핑하느냐"뿐.**

| 비교 항목 | 처음부터 네이티브 (Capacitor day 1) | PWA 먼저 → 전환 |
|---|---|---|
| 개발 속도 | 매 수정마다 Xcode 빌드→기기 설치/TestFlight 업로드 대기. **Mac 필수**(원격 개발 환경에선 CI 우회 필요) | **수정 → 푸시 → 폰에서 URL 새로고침.** 마찰 0 |
| 초기 비용 | 첫날부터 연 $99 | $0. 계속 쓸 앱인지 검증된 뒤에 지출 |
| 배포·수정 반영 | TestFlight 빌드 처리·심사 대기 | 즉시 |
| STT(회화 말하기) | 네이티브 음성인식 무료 사용 | 폴백(녹음+Gemini 전사, 분당 $0.002)로 해결 — 어차피 회화는 LLM 호출 필요라 자연스러움 |
| 푸시 알림 | 안정적(APNs) | 가능(iOS 16.4+ 웹푸시), 신뢰성은 APNs보다 낮음 |
| 위젯 | 가능 | **불가** — 유일한 실질 결손 |
| 백그라운드 오디오(흘려듣기) | 안정적 | 대체로 동작, 간헐적 끊김 가능 |
| 데이터 안정성 | 앱 저장소 | localStorage 퇴거 리스크 → 백업/복원 기능으로 방어(JP 앱에 이미 있음) |
| 전환 비용 | — | Capacitor 래핑 + 푸시 APNs 교체 + 위젯 + IAP ≈ **1~2주** |

**판정: PWA 먼저.** 근거:

1. 이 프로젝트의 최대 리스크는 기술이 아니라 **"내가 계속 쓰는 앱인가"(리텐션)**. 검증을 가장 빨리 시작하는 경로가 PWA다.
2. 원격 개발(Claude) + 아이폰 확인 루프에서 PWA만 마찰이 없다. 네이티브 루프는 Mac/CI/TestFlight 대기가 매 수정마다 붙어 1개월 타임라인이 깨진다.
3. 처음부터 네이티브로 가서 아끼는 것은 전환 작업 1~2주뿐인데, 그 대가로 개발 기간 내내 느린 루프 + 첫날 $99 + Mac 의존을 진다.
4. PWA 제약 중 진짜 불가는 위젯 하나이고, 위젯은 v1 핵심 기능이 아니다(성장 스테이지 에셋 재사용으로 전환 시 저비용 추가).

**처음부터 네이티브가 정당화되는 조건**(해당 시 재검토): 위젯이 핵심 훅일 때 · 스토어 판매가 이미 확정일 때 · 흘려듣기(백그라운드 오디오)가 주 사용 시나리오의 절반 이상일 때.

전환 트리거: 1개월 완성 후 **2~4주 실사용에서 주 4일 이상 쓰게 되면** Capacitor 래핑 + TestFlight 착수(COMMERCIAL.md 권고 경로와 동일).

## 6.6 백엔드 설계 (웹앱 단계 조기 도입분 + 상용화 확장) `[설계만 · 2026-07-15]`

> **전제 전환**: 기존 6.5는 "PWA 먼저 · 백엔드는 상용화 때(COMMERCIAL 부록 A)"였다. 그런데 **웹앱 단계에서도 백엔드/DB가 필요한 기능이 이미 들어왔다** — ① 소셜 로그인(구글·카카오) ② 친구 초대·소셜 뱃지 ③ 기기 간 학습 데이터 동기화 ④ 결제(구독). 순수 1인 오프라인 앱이면 localStorage로 충분하지만, 위 4개 중 하나라도 켜는 순간 서버가 사실상 필수다. 그래서 **백엔드를 "상용화 전용"에서 "웹앱 단계에 점진 도입"으로 앞당겨 설계**한다(전면 재작성이 아니라 오프라인-우선 위에 얹는 방식).

### 6.6.1 도입 판단 — 무엇이 진짜 서버를 요구하나

| 기능 | localStorage로 가능? | 서버 필요 이유 |
|---|---|---|
| SRS·스트릭·포인트·뱃지 진행값 | 가능(현재) | 기기 바꾸면 유실 · 백업 위해서만 서버 |
| **소셜 로그인**(구글/카카오) | 불가 | OAuth 토큰 검증·세션은 서버(또는 Auth 서비스) 몫 |
| **친구 초대·소셜 뱃지**(4.10) | 불가 | 사용자 간 관계·초대 상태는 공유 저장소 필요 |
| **기기 간 동기화** | 불가 | 서버가 진실의 원천(source of truth) |
| **결제·구독 상태** | 불가 | 영수증 검증·엔타이틀먼트는 반드시 서버(위변조 방지) |
| LLM/TTS 키 은닉·사용량 캡 | 프록시만 | 키를 클라이언트에 두면 유출 — 서버 함수 필수(이미 `api/proxy.js` 예정) |

**결론**: 로그인·친구·결제를 붙이는 순간이 백엔드 도입 시점. 뱃지의 `친구 초대`·`자립 회화` 카테고리와 홈 백분위(상위 %) 등은 이때 더미→실데이터 전환.

### 6.6.2 권장 스택 — Supabase 단일 (COMMERCIAL 부록 A와 일치)

- **Supabase Pro(월 $25)**: Postgres(DB) + Auth(소셜 로그인) + Edge Functions(서버 함수) + Storage + RLS(Row Level Security)를 한 곳에서. 부록 A가 이미 이걸 골라뒀으므로 웹앱 단계와 상용화가 같은 스택을 공유(리팩터링 최소).
- **오프라인-우선(offline-first)**: 클라이언트는 계속 localStorage/IndexedDB를 1차 캐시로 쓰고, 로그인 시 Supabase와 **동기화**. 비로그인도 앱은 그대로 동작(게스트 모드) → 로그인하면 그 시점 로컬 데이터를 서버로 업로드·병합.
- LLM/TTS 프록시(`api/proxy.js`)는 Vercel 서버리스로 시작하되, 사용량 카운트가 필요해지면 Supabase Edge Function으로 이전(DB에 사용량 기록).

**DB 선택 — SQLite vs PostgreSQL 검토 결론(2026-07-15): Postgres.** 근거:

| 기준 | SQLite | PostgreSQL |
|---|---|---|
| 동시 쓰기 | 단일 writer 락 — 다수 사용자 동시 저장에 취약 | 다중 사용자 동시성에 강함 |
| 관계형(친구·추천·대결·구독) | 조인·제약 직접 관리 부담 | 관계·제약·트랜잭션 네이티브 |
| 사용자 격리(보안) | 앱 레벨에서 수동 | **RLS**로 행 단위 자동 |
| 성장 대응 | 단일 파일 → 스케일 한계(Turso/D1 필요) | 수직·수평 확장 경로 명확 |

- "사용자 많아지면 데이터가 쌓인다"는 우려가 **오히려 Postgres를 택하는 이유** — 다중 사용자·동시 접근·관계형(특히 친구·주간 대결·추천)이 SQLite의 약점이다.
- **클라이언트엔 이미 로컬 저장(IndexedDB/localStorage)이 있으므로 클라이언트용 SQLite는 불필요.** 서버 Postgres는 동기화·소셜·결제의 진실 원천 역할만.
- **DB 저장 자체는 스케일 병목이 아니다** — 사용자당 텍스트(단어장·SRS·뱃지)는 수 KB~수 MB라 Supabase 무료 500MB에도 수천 명. 실제 비용 드라이버는 LLM/TTS 호출(3장·COMMERCIAL 2장). 누적 로그(대화·사용량)만 TTL/롤업으로 관리.
- **SQLite 계열이 후보가 되는 유일 경로**: 스택을 Cloudflare로 통일할 때의 **Cloudflare D1**(엣지 SQLite) 또는 **Turso(libSQL)** — 서버리스에서 도는 SQLite라 동시성 단점을 상당 부분 보완. 단 친구·구독처럼 관계·RLS가 핵심이면 Postgres가 여전히 단순. → **기본은 Supabase Postgres, 스택 통일 시 D1/Turso를 실질 대안으로 열어둠.**

### 6.6.4 인증 — 구글 · 카카오 로그인

- **구글**: Supabase Auth 기본 지원(OAuth Provider 등록만).
- **카카오**: Supabase가 기본 프로바이더로 두지 않으므로 **커스텀 OAuth/OIDC 연동**(카카오 REST 앱 키 + Edge Function으로 토큰 교환) 또는 Kakao JS SDK 로그인 후 Supabase 세션 발급. 국내 타깃이라 카카오는 사실상 필수.
- **애플**: 스토어 출시 + 소셜 로그인 제공 시 **Sign in with Apple 동반 필수**(App Store Guideline 5.1.1 / D.5). 웹앱 단계에선 선택, 네이티브 전환 시 추가.
- 게스트→로그인 승격: 로컬(익명) 데이터를 최초 로그인 시 그 계정으로 이관(1회 병합).

### 6.6.5 보안

- **RLS(Row Level Security) 필수**: 모든 테이블에 `user_id = auth.uid()` 정책 — 남의 행 접근 원천 차단. 소셜/친구 테이블은 양방향 관계만 읽기 허용.
- **API 키는 서버에만**: Gemini/TTS/카카오 시크릿은 Edge Function 환경변수. 클라이언트 번들에 절대 포함 금지.
- **사용량 캡·레이트리밋**: `usage` 테이블 카운트로 무료 티어 상한(2.3 LLM 최소화 원칙과 연동) — 폭주/어뷰징 방어.
- **입력 검증·서버 권위**: 뱃지 획득·포인트 적립·결제 상태는 **서버가 판정**(클라이언트가 "나 달성함" 보내는 걸 그대로 믿지 않음). 특히 결제 엔타이틀먼트는 영수증 서버 검증만 신뢰.
- **개인정보·계정 삭제**: 인앱 계정 삭제 API(App Store 5.1.1 필수) — `profiles`·연관 데이터 캐스케이드 삭제. 최소 수집(이메일·표시명 정도).

### 6.6.6 결제 — 채널별 정리(웹 vs 스토어의 긴장)

| 채널 | 결제 수단 | 정기결제 | 수수료 | 비고 |
|---|---|---|---|---|
| **웹(PWA)** | 카카오페이 · 토스페이먼츠 · 구글페이(웹) | 빌링키 기반 정기결제(국내 PG) | PG ~3% | 스토어 밖이라 IAP 강제 없음. 국내 타깃엔 카카오페이 유리 |
| **iOS(App Store)** | **Apple IAP 강제** | StoreKit 구독 | 15%(Small Business) | 디지털 구독은 외부결제 불가(Guideline 3.1.1 / D.2) |
| **Android(Play)** | **Google Play Billing** | Play 구독 | 15% | 동일 원칙 |

- **웹앱 단계 권장**: 우선 **카카오페이/토스 정기결제**로 시작(국내 사용자·낮은 진입장벽). 스토어 출시 시 **IAP/Play Billing 병행**하고 **RevenueCat**으로 영수증 검증·구독 상태를 `subscriptions` 테이블에 일원화(부록 A와 동일).
- **엔타이틀먼트 단일화**: 어느 채널로 결제했든 서버의 `subscriptions.status`가 유일한 판단 기준 → 클라이언트는 그 값만 보고 프리미엄 기능 잠금 해제. 구매 복원(Restore) 필수(D.2).

### 6.6.7 단계적 도입 로드맵

| 단계 | 범위 | 트리거 |
|---|---|---|
| **P0 (현재)** | 전부 localStorage/IndexedDB · 게스트 · 프록시(키 은닉)만 | — |
| **P1 인증+동기화** | Supabase Auth(구글·카카오) + `learning_state`·`badges` 동기화 | "기기 바꿔도 유지"가 필요해질 때 |
| **P2 소셜** | `friends`·`invites`·`referrals`·`duels` + 친구·초대·**주간 대결**·소셜 뱃지·백분위 실데이터 | 친구 기능/랭킹을 켤 때 |
| **P3 결제** | `subscriptions` + 카카오페이/토스 정기결제(웹), 이후 IAP/RevenueCat(스토어) + **추천 할인** 반영 | 유료화 결정 시 |

각 단계는 앞 단계 위에 얹기만 하면 되도록 스키마를 미리 잡아둔다(P0에서도 데이터 모델은 P1 스키마와 호환되는 형태로 유지 — 예: 뱃지 진행값 `{cat_key, tier, dates}` 구조 그대로).

### 6.6.8 소셜 기능 설계 — 친구 · 추천 할인 · 주간 대결 `[설계만 · P2]`

> 전부 다중 사용자·관계형·(대결은)실시간 성격 → **서버 DB(Postgres) 없이는 불가**. 6.6.3 `friends`/`referrals`/`duels`/`duel_events` 테이블이 이들의 저장소.

- **친구**: 초대 코드/링크(`invites`)로 연결 → 수락 시 `friends` 양방향 엣지. 친구 목록에서 상대 프로필(레벨·스트릭·주간 점수) 열람. 뱃지 `친구 초대` 카테고리(4.10)가 실데이터로 전환.
- **친구 추천 할인**: 추천인 코드로 가입/결제 시 `referrals` 기록 → **양쪽에 할인/혜택 지급**(예: 추천인·피추천인 각 1개월 할인 또는 포인트). **어뷰징 방지**가 핵심 — 피추천인이 실사용(온보딩 완료·최초 결제 등) 조건을 충족해야 `reward_status=granted`로 전환, 지급은 **서버가 판정**(클라이언트 신뢰 금지). 할인은 `subscriptions.discount_ref`로 결제에 반영.
- **친구와 주간 대결(애플워치 피트니스형)**: 친구 1:1로 **한 주(ISO week) 동안 학습 점수 겨루기**. 점수원 = 학습 활동(`duel_events`: 단어카드 정답·듣기 세션·회화 완주 등에 가중치). 주말에 `duels.winner` 확정 → 승패 기록·연승 스트릭. UI는 상대와 나의 주간 점수 바 + 남은 시간(애플워치 "친구와 겨루기" 레이아웃 차용). **뱃지 후보 카테고리 신설**: `주간 대결 승리`(누적 승수 티어) — 4.10 카테고리에 추가 여지(현재 9종 → 확장 범위 8~12 안에서).
  - 공정성·부정 방지: 점수는 **서버가 학습 로그를 집계**(클라이언트가 보낸 raw 점수 불신), 비정상 급증은 사용량/레이트 체크로 필터. 상대가 비활성이면 "혼자 목표" 모드로 폴백.
- **실시간성**: 주간 대결 점수판은 Supabase Realtime(Postgres 변경 구독)으로 갱신 가능 — 다만 P2 초기엔 주기적 폴링/새로고침으로도 충분(실시간은 후순위 최적화).
