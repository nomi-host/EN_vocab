# 상용화(판매) 검토 — 저작권 · 수익성

작성일: 2026-07-11 · 전제: 연 $99 Apple 개발자 계정 결제 후 스토어 판매

**표기 정책**: 법적 리스크 제거를 위해 이 문서(및 코드·다른 문서 전체)에서 타사 서비스명은 특정하지 않고 "선행 어휘앱", "선행 AI 회화앱 A/B/C" 등 카테고리 표현으로 익명화한다. 실제 서비스명이 필요하면 대화로 별도 요청.

## 1. 저작권 / 라이선스

### 1.1 결론 요약

**판매 가능하다. 단, 개인용 기획(PLAN.md)의 데이터 소스 중 3개를 교체해야 한다.**

| 소스 | 개인용 | 판매 시 | 조치 |
|---|---|---|---|
| Oxford 3000/5000 | 사용 가능 | **불가** — 약관상 "domestic and private use only", 상업 이용은 OUP 별도 라이선스 계약 필요 | **NGSL(CC BY 3.0) + 자체 빈도 기반 레벨 밴드로 교체** |
| Datamuse API | 사용 가능 | **불가** — 무료는 비상업 한정, 상업용은 유료 계약 | **WordNet 셀프호스트로 교체** (Princeton 라이선스, 상업 OK, 출처 표기) |
| Merriam-Webster API | 사용 가능 | **불가** — 무료 키는 비상업 한정 | 제거 (기능상 대체 가능) |
| Free Dictionary API / Wiktionary 계열 | 가능 | 가능 (조건부) | 데이터가 CC BY-SA → **출처 표기 + 데이터 부분 동일조건 유지**. 무료 호스팅 서비스는 SLA 없음 → kaikki.org(wiktextract) 데이터 셀프호스트 권장 |
| etymology-db | 가능 | 가능 (조건부) | 위와 동일 (Wiktionary 파생, CC BY-SA) |
| Tatoeba 예문 | 가능 | 가능 | CC BY, 출처 표기 |
| WordNet | 가능 | 가능 | 상업 OK, 라이선스 고지 동봉 |
| Google TTS / Gemini | 가능 | 가능 | 종량제 상업 이용 허용. **단 Gemini 무료 티어는 입력을 구글이 학습에 사용 → 사용자 데이터가 들어가는 상용 서비스는 유료 티어 필수** |
| LLM 생성 콘텐츠(한국어 뜻/예문/첨삭) | 가능 | 가능 | Gemini 약관상 출력물 상업 이용 허용 |

### 1.2 데이터 외 체크포인트

- **선행 어휘앱(국내 대표) UI 모방**: 학습 방식(예문 빈칸 입력) 같은 아이디어 자체는 저작권 보호 대상이 아니지만, 화면 구성·문구·그래픽을 그대로 옮기면 **부정경쟁방지법·저작권 분쟁 소지**. 판매 버전은 구조만 참고하고 비주얼은 독자 디자인으로.
- **실존 브랜드 언급**: 페르소나의 "픽코마 다니는 개발자" 등 실제 상호는 **가상 회사명으로 교체** (상표 이슈).
- **아바타 이미지**: 생성형 도구로 만들 경우 해당 도구의 **상업 이용 허용 여부** 확인. 실존 인물 닮은 이미지 금지.
- **폰트/아이콘**: 상업 배포 허용 라이선스(OFL 등)만 사용.
- **국내 판매 행정**: 사업자등록 + 통신판매업 신고, 개인정보처리방침·이용약관 필요 (유료앱/구독이면 사실상 필수).

### 1.3 교체 비용

NGSL·WordNet·wiktextract 모두 오픈 데이터라 **추가 금전 비용은 0원**. 대신 시드 데이터 재가공(빈도 기반 레벨 밴드 산정, WordNet 유의어망 굽기) 작업이 1~2주분 추가된다.

## 2. 수익성

### 2.1 비용 구조 (구독 월 5,000원 기준 — 저가·심플·캐릭터 포지셔닝)

| 항목 | 금액 (사용자 1인/월) |
|---|---|
| LLM (회화·첨삭, Gemini Flash 유료 티어) | 라이트 ~500원 · 헤비 ~4,000원 |
| TTS (무료 티어 초과분) | 0~1,500원 |
| 앱스토어 수수료 (Small Business 15%) | 750원 |
| **인당 마진** | **캡 없으면 헤비유저는 마진 ~0원(적자 근접)** → 사용량 캡 적용 시 약 2,200~3,700원 |

- **월 5,000원 가격에서는 사용량 캡이 선택이 아니라 필수.** 캡 예시: 회화 하루 30턴 + TTS 캐시 강제 → 인당 변동비 상한 ~2,000원. 캡 초과분은 다음날 리셋(선행 어휘앱(국내 대표)·선행 AI 회화앱 A도 동일한 세션 제한 구조).
- 참고: 애플 IAP는 임의 가격이 아니라 가격 티어에서 선택 — 실제로는 4,900~5,500원대 티어로 확정하게 됨.
- 고정비: Supabase 월 $25 + 개발자 계정(첫해 $124, 이후 연 $99) ≈ 월 5만 원.
→ **손익분기: 유료 구독자 약 15~20명.** 100명이면 월 ~20만 원, 500명이면 월 ~110만 원 수준의 순수익.

### 2.2 구조적 리스크

1. **판매 순간 백엔드가 필수가 된다.** 현재 설계(localStorage + 공용 프록시)로는 판매 불가 — 계정/동기화/결제/사용량 제한 서버 개발이 실질 최대 비용(돈보다 시간).
2. **LLM 변동비 때문에 "일회성 판매(buy once)"는 위험.** 헤비 유저 1명이 평생 API 비용을 잠식한다. → **구독 모델 + 사용량 캡**(선행 어휘앱(국내 대표)도 세션 제한제)이 사실상 강제.
3. **어뷰즈 방지 필수.** 내 API 키로 도는 회화 기능은 캡 없이는 적자 전환 가능.

### 2.3 시장성 판정

- 시장은 **검증된 레드오션**: 선행 어휘앱(국내 대표)·선행 AI 회화앱 A·선행 회화앱 B·선행 게임화 학습앱·선행 회화앱 C가 마케팅에 대규모 투자 중. 오가닉만으로 유료 전환 확보는 어렵다.
- 현실적 시나리오: 마케팅 없이 커뮤니티/입소문 기반 **월 구독자 수십 명 = 월 수십만 원(부업 규모)**. $99 회수는 쉽지만 그 이상은 마케팅 투자에 비례.
- 차별화 여지: 범용 앱과 정면승부 대신 **니치 특화** — 예: 어원/어근 타고타고 학습, 개발자·직장인 영어 특화, 페르소나 커스텀. 범용 기능은 대형 앱이 항상 더 잘한다.

### 2.4 경쟁 앱 현황 (2026-07 서치)

"완전 동일" 앱은 없으나, 기획의 세 축 각각에 선점자가 존재:

| 축 | 기존 앱 | 비고 |
|---|---|---|
| 단어 암기 (예문 빈칸·음성·SRS) | **선행 어휘앱(국내 대표)** | 거의 동일 구조, 국내 지배적 |
| 페르소나 AI 회화 | **선행 아바타 튜터앱**(아바타 튜터), **선행 AI 회화앱 A**(롤플레이·프리토킹, 월 2만원대) | 단 선행 AI 회화앱 A은 500일 커리큘럼이 중심, AI 대화는 부가 기능. 기존 앱의 AI는 전부 "튜터" 역할 — **관계 기반 페르소나(애인·불친절한 직원 등) 중심 + 피드백 게이트 + SRS 연동 설계는 부재** |
| 어원/어근 학습 | **해외 어원학습 서비스 A**(어근·어원 분해), **해외 어원학습 서비스 B**(SRS+AI) | 영어권용. **한국어 UI 앱은 부재** |
| 작문 첨삭 | 선행 AI 회화앱 A 등에 부분 포함 | 단독 강자 없음 |

- 빈자리: **세 축 통합 + 어원 타고타고 + 수능세대 산출 특화의 한국어 앱은 없음.**
- 단, 빈 조합 = 미검증 수요. 사용자에겐 "선행 어휘앱(국내 대표)+선행 AI 회화앱 A 2개 구독"이라는 대체재가 이미 있음 → 세일즈 포인트는 "두 구독 합산(월 4~5만 원)보다 싸고, 단어-회화가 연동된다"로 잡아야 함.
- 결론: 표절 시비 걱정 없이 개발 가능. 수익 기대치는 2.3 판정(부업 규모) 유지.

### 2.5 권고 경로

1. **P1~P5를 개인용으로 완성** (현 기획 그대로, Oxford/Datamuse 사용 무방)
2. TestFlight로 지인 소규모 배포 → 리텐션 확인 (여기까지 $99 외 추가 비용 없음)
3. 반응이 실재할 때만 상용화 리팩터링: 1.1 데이터 교체 + 계정/결제 백엔드 + 사용량 캡 + 구독 모델
4. 판매 시작 전 1.2 행정·법무 체크리스트 처리

**요약: 라이선스는 교체로 해결 가능, 손익분기는 낮음(10~20명), 그러나 기대 수익은 부업 규모. "완성 → 소규모 검증 → 반응 있으면 전환"이 합리적이며, 처음부터 판매를 전제로 설계를 바꿀 필요는 없다.**

## 부록 A. 백엔드 구축 비용

판매 전환 시 필요한 것: 계정(인증) · 서버 동기화 · 결제(구독) · 사용량 캡. 직접 개발(+AI 코딩) 전제.

### A.1 권장 스택 — "월 $25 고정" 구성

| 구성요소 | 선택 | 비용 |
|---|---|---|
| 인증 + DB + 서버함수 | **Supabase Pro** (Postgres, Auth, Edge Functions 통합) | **월 $25** |
| LLM/TTS 프록시 + 사용량 캡 | Supabase Edge Functions로 기존 proxy.js 이전, 사용량 카운트는 DB | 위에 포함 |
| 구독 결제 | **IAP(앱 내 구독) + RevenueCat** (영수증 검증·구독 상태 관리 대행) | 월매출 $2,500까지 무료 |
| 정적 호스팅 | Cloudflare Pages 등 | $0 |

- **주의: 현 Vercel 무료(Hobby) 플랜은 상업 서비스 이용 불가** → 판매 시 Vercel Pro(월 $20)로 올리거나 위처럼 Supabase/Cloudflare로 이전.
- IAP만 쓰면 국내 PG(토스페이먼츠 등) 계약 불필요. 웹 결제 병행 시 PG 수수료 ~3% + 가입 절차 추가.
- 규모별: 구독자 ~100명까지 월 $25 그대로. ~500명이라도 인프라는 월 $50~100 (지배적 비용은 인프라가 아니라 인당 LLM 변동비 — 2.1 참조).

### A.2 개발 기간 (실질 비용은 돈보다 시간)

| 작업 | 기간 |
|---|---|
| 계정 + 로컬→서버 동기화 (localStorage 구조를 계정 기반으로 마이그레이션, 기기 간 병합) | 2~3주 (최대 난관) |
| 사용량 캡·어뷰즈 방지 (일일 회화 턴/TTS 한도) | 3~5일 |
| IAP 구독 + RevenueCat 연동 + 페이월 | 1~2주 |
| 앱 심사 요건 (약관·개인정보처리방침·계정 탈퇴·복원 구매) | 2~3일 |
| **합계** | **약 4~7주** |

외주 시세로 환산하면 1,500만~3,000만 원 상당. 직접 개발이면 현금 지출은 **월 $25 + 개발자 계정 연 $124**가 전부.

## 부록 B. UI 차별화 — "선행 어휘앱(국내 대표) 따라했다" 소리 안 나오게

법적 기준: 아이디어(빈칸 학습, SRS, 딕테이션)는 보호 대상이 아니고, **"표현"(레이아웃·색·문구·아이콘 조합)의 실질적 유사성**이 문제가 된다. SRS·빈칸 채우기·잔디는 오픈소스 암기카드 앱·게임화 학습앱·코드 저장소 서비스부터 있던 업계 공통 요소라 기능 자체는 안전.

### B.1 선행 어휘앱(국내 대표)의 시그니처 (피해야 할 조합)

1. 다크 배경 + 보라/청록 포인트 컬러
2. 카드 2단 구성: 상단 한국어 문장(녹색 강조) / 하단 영어 문장의 **어두운 알약(pill) 모양 빈칸**
3. "6분 뒤 복습" 청록 알약, 레벨 뱃지 + 물음표, 트로피 진행바
4. 정답 후 **아이콘 5개 가로 행** (다른 예문 · 오답노트 · 말해보기 · 단어정보 · AI질문)
5. 하단 3버튼: x0.8 듣기 / 다시 듣기 / 다음 문제
6. 레이블 조합: "음성 모드", "길이 보기" 등

→ 위 요소를 **한 화면에 3개 이상 재현하지 않는 것**이 실무 기준. 스크린샷을 나란히 놓고 제3자가 "같은 앱 계열"로 혼동할 정도면 위험.

### B.2 조정 전략 (기능은 유지, 표현만 교체)

1. **비주얼 계보를 JP 앱에서 상속** — 가장 쉬운 해법. 라이트 모드 + 모노톤 + 포인트 1색(iOS 기본앱 톤매너)은 선행 어휘앱(국내 대표)와 정반대 디자인 언어이고, "우리 앱 패밀리 디자인"이라는 독자적 근거가 생긴다.
2. **빈칸 표현 교체**: 어두운 pill 대신 밑줄(______) 또는 점선 박스. 힌트도 "길이 보기" 대신 첫 글자 공개 방식으로.
3. **정답 후 흐름 재구성**: 아이콘 5개 행 대신 **바텀시트로 단어 상세 미리보기**(유의어·어원 칩 포함)를 올리고 그 안에 액션 배치 — 이 앱의 강점(타고타고)을 정답 직후에 노출하는 구조라 차별화와 학습 효과를 동시에 챙긴다.
4. **진행 표현 교체**: 트로피 진행바 대신 숫자 카운터/링. "6분 뒤 복습" 알약 대신 다음 복습 시점을 카드 하단 텍스트로.
5. **레이블 독자화**: 기능 이름을 새로 짓는다 (음성 모드 → 말하기 입력, 오답 노트 → 복습 노트 등). 문구 조합이 그대로면 그것만으로 "봤다"는 인상을 준다.
6. **차별 기능을 첫인상에 배치**: 선행 어휘앱(국내 대표)에 없는 어원/어근 네트워크, 페르소나 회화를 홈과 카드 직후 화면의 중심에 두면 비교 프레임 자체가 사라진다. 단어 상세의 "같은 어원 타고타고"가 이 앱의 시그니처 화면이 될 수 있다.

와이어프레임(docs/wireframe.html)은 구조 설계용이므로, 디자인 단계에서 위 기준으로 카드 화면(2·3번 화면)을 우선 재해석할 것.

## 부록 C. 세무 처리 (기존 디자인업 사업자 활용)

- **새 사업자 불필요** — 기존 사업자에 홈택스 정정신고로 **업종 추가**: 응용 소프트웨어 개발 및 공급업(722000 계열). 무료. 판매 시작 시점에 통신판매업 신고 추가.
- **비용**: 개발자 계정·서버·API·피그마 등을 필요경비로 공제. 해외 결제는 카드 명세·인보이스 증빙 보관.
- **초기 절세 효과**: 앱 사업 초기 결손금은 **디자인업 소득과 통산** → 앱 비용만큼 기존 소득 과세표준 감소.
- **소득**: 앱스토어 정산금을 동일 사업자의 사업소득으로 합산 신고 (종합소득세 6~45% 누진).
- **세무사 확인 필요 2건**: ① 애플(해외 법인 지급) 정산금의 부가세 영세율 적용 여부 (구글플레이는 별도 확인) ② 기장 방식 — 단순경비율 vs 간편장부(실제 경비), 간이/일반과세 여부.

## 부록 D. App Store 승인 요건 (수익화 게이트 — 후반부이나 사활적)

앱 자체를 아무리 잘 만들어도 **심사 거절이면 수익화가 0**이므로, 아키텍처 결정 단계부터 아래를 전제로 설계한다. (2026 기준 App Review Guidelines)

### D.1 최대 리스크 — Guideline 4.2 "최소 기능" (웹뷰 래퍼 거절)

우리는 Capacitor(웹뷰) 전환을 계획하므로 **이게 1순위 리스크**. Apple은 "웹사이트를 감싼 껍데기"를 거절한다. 통과 조건 = 네이티브 레이어를 실제로 갖출 것:

- **푸시 알림** — Apple이 "웹과 앱을 가르는 1순위 차별점"으로 본다. (기획 4.9에 이미 포함) → 반드시 네이티브 푸시로.
- **오프라인 상태 화면** — 비행기 모드에서 흰 화면/브라우저 에러가 뜨면 즉시 웹래퍼로 판정된다. → PWA 캐시 + 네이티브 오프라인 화면.
- **OS 통합** — 위젯(기획 4.9), 잠금화면 오디오 컨트롤(흘려듣기) 등.
- 네이티브 탭바 등 앱다운 셸.
- **결론: "푸시 + 오프라인 + 위젯"은 선택 기능이 아니라 심사 통과용 필수.** 출시 빌드에 반드시 포함.

### D.2 결제 (Guideline 3.1.1)

- 디지털 구독은 **반드시 Apple IAP** 사용(외부 결제 불가) → Small Business 15% 수수료(부록 A 전제와 일치).
- 결제 전 **가격·갱신 주기·취소 방법 전체 고지** 필수. 구매 복원(Restore) 기능 필수.

### D.3 계정·개인정보 (Guideline 5.1.1)

- **인앱 계정 삭제 필수** — 계정 생성 기능이 있으면 앱 안에서 삭제 가능해야 함(백엔드에 삭제 API 필요, 부록 A 범위).
- 개인정보처리방침 링크를 App Store Connect + 앱 내 접근 쉬운 곳에 배치. 개인정보 수집 라벨(Privacy Nutrition Label) 작성.

### D.4 AI · 대화(회화) 기능 — Guideline 1.2 + AI 고지

우리 앱의 페르소나 회화·작문 첨삭은 **외부 AI(Gemini) + 사용자 생성/AI 생성 콘텐츠**라 심사에서 별도로 본다:

- **외부 AI 사용 고지 + 사용자 동의** 필요.
- **콘텐츠 조정 4종 세트 필수**: ① 부적절 콘텐츠 필터링 ② 신고 기능(적시 대응) ③ 사용자/콘텐츠 차단 ④ 연락처 공개.
- **사용자 추가 페르소나(기획 4.6)**: 자유 텍스트를 프롬프트에 직접 넣지 않고 정형 필드로만 받는 현재 설계가 **심사·안전 양쪽에 유리** — 유지.
- LLM 출력 세이프티 필터(욕설·성적·폭력) 프록시 단에서 적용.

### D.5 기타

- **최신 SDK 빌드 요구**(2026.4부터 iOS 26 SDK 이상). 소셜 로그인 제공 시 Sign in with Apple 동반 필요.

### D.6 백엔드/DB가 승인과 직결되는 지점 (부록 A와 연계)

| 승인 요건 | 필요한 백엔드/DB 작업 |
|---|---|
| 인앱 계정 삭제(D.3) | 계정 테이블 + 하드 삭제 API + 관련 학습데이터 파기 |
| IAP 구독(D.2) | RevenueCat 영수증 검증 + 구독 상태 DB |
| 콘텐츠 신고·차단(D.4) | 신고 레코드 테이블 + 차단 목록 + 운영자 확인 경로 |
| AI 세이프티(D.4) | 프록시 단 모더레이션 필터 + 로깅 |
| 개인정보(D.3) | 수집 항목 최소화 + 처리방침과 실제 저장 일치 |

**로드맵상 위치**: 개인용 완성(P1~P5) → 소규모 검증 → **상용화 리팩터링 시 부록 A(백엔드) + 부록 D(승인 요건)를 한 묶음으로** 처리. 승인 요건은 아키텍처에 스며들어야 하므로, 판매를 결정하면 D.1(네이티브 레이어)·D.4(모더레이션)를 백엔드 설계 초기에 반영한다.
