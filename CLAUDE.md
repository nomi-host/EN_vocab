# 작업 메모리 — EN_vocab / NoEnglish (한→영 어휘 학습 PWA)

## 작업 원칙 (★가장 중요★)
- **수정할 때 내 마음대로 판단해서 사용자의 디자인을 롤백/제거하지 말 것.**
  무언가 바꾸기 전에 반드시 **기획서(`docs/PLAN.md`)와 파이프라인(`pipeline/`)을 먼저 확인**하고 기존 설계를 존중한다.
  임의 판단으로 되돌렸다가 다시 만드는 것은 **토큰·시간 낭비**다. (예: 홈 뱃지 트랙을 마음대로 제거했다가 다시 복원한 사례 — 반복 금지.)
- 요청받은 것만 정확히, **하나씩 실제 브라우저 렌더로 검증**하고 넘어간다. "됐다"고 말하기 전에 반드시 렌더 확인.
- 애매하면 추측해서 큰 변경을 하지 말고 먼저 확인한다.

## 대화 스타일
- 사용자와 대화할 때는 항상 **존댓말**을 사용한다.

## 디자인 규칙
- **이 앱에는 절대 이모지를 넣지 않는다.** (픽토그램/컬러 이모지 금지. UI 텍스트·안내문·알림 어디에도.)
  - 체크(✓)·닫기(✕)·화살표(→ › ↔) 같은 기능적 텍스트 글리프는 예외.
- **섹션 타이틀은 `.sec` 클래스(26px/800) 공용 — 새 화면 만들 때도 이거 재사용.** (2026-07-15, "SOS"/"뱃지 컬렉션" 등 타이틀이 13px로 너무 작다는 지적으로 2배 확대.)

## 문구 규칙
- 안내(업데이트 안내 등) 문구는 핵심/중요 부분을 볼드로 강조한다. (`WHATS_NEW`는 `**중요부분**` 마크업, `renderBold`가 `<b>`로 렌더.)

## 버전 표기
- 기능을 수정/배포할 때마다 `index.html`의 `APP_VERSION`을 갱신하고, `<head>`의 `?v=` 캐시 무효화 태그(manifest/words.js/rat_run.js)도 **같은 버전으로 함께** 갱신한다.
- `APP_BUILD`는 작업 당일 날짜(YYYY-MM-DD).
- **`version.json`의 `v` 값도 APP_VERSION과 항상 같이 갱신할 것.** PWA는 새로고침 제스처가 없어서, 앱이 이 파일을 캐시 없이 주기적으로 읽어 배포 감지 배너(`.update-banner`)를 띄운다 — 안 올리면 사용자에게 새 버전 알림이 안 감.

## 데이터 파이프라인 메모
- 시드는 `pipeline/`에서 생성 → `pipeline/merge_cefrj.js`가 `words.js`로 병합. `words.js`는 직접 수정 금지(배치/보강 파일 수정 후 재빌드).
- 다의어 senses는 `pipeline/senses_parts/*.json`(에이전트별 분할)을 merge가 합침. WordNet 뼈대는 `pipeline/wordnet_senses.json`.
- **다의어 배치 실행 개수 제한**: 한 세션에서 배치를 **2개 완료하면 자동으로 다음 배치를 이어서 실행하지 말 것.**
  사용자가 "더 해" 등으로 명시적으로 요청하기 전까지는 멈추고 기다린다. (세션 리밋 대비 보수적 실행 원칙과 별개로,
  배치 개수 자체를 사용자 통제하에 둔다.)
