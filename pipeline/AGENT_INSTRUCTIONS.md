# 다의어(senses) 작성 에이전트 지침

"NoEnglish"(한→영 어휘 학습 PWA, /home/user/EN_vocab)의 표제어별 다의어 데이터를 작성한다.
포맷·방식은 파일럿으로 이미 검증됨. 배정된 배치를 고품질로 작성하는 것이 임무.

## 입력
배정 파일: `/home/user/EN_vocab/pipeline/work/<BATCH>.json`
= 배열 `{ "word", "ko"(현재 단일 뜻), "pos"(배열), "wn"(WordNet 의미 목록: {pos,def,syn} 배열) }`.

## 방식 (하이브리드: WordNet 근거 구조 + 한국어 작성)
각 단어에 `senses[]` 작성. `wn`(WordNet)을 "어떤 의미가 존재하는지"의 **근거**로 사용 — 누락·창작 방지.
단 WordNet은 과분할이고 정의가 기술적이므로:
1. **병합**: 과분할된 synset을 학습자용 의미로 합침(예: "light"는 WordNet 48개 → 학습자용 ~5개: 빛/전등/가벼운/연한/불붙이다).
2. **선별**: 학습자가 실제로 필요한 의미만. 희귀·고어·전문 뜻은 버림. 실사용 빈도순, 흔한 것 먼저(lv 오름차순).
3. **품사 분리**: 여러 품사로 쓰이면 의미 분리(light = 빛(명사)/가벼운(형용사)/불붙이다(동사)).
4. 학습자 기준 정말 단일 의미면 **1개만**. 억지로 늘리지 말 것 — 양보다 정직.
5. 항목의 `ko`가 현재 대표 뜻 → 첫 번째 sense가 이와 일치하도록.

## 출력 스키마 (sense 하나당) — 파일럿과 정확히 일치시킬 것
먼저 `/home/user/EN_vocab/pipeline/senses_parts/00_pilot.json`에서 "run","light","fair"를 읽고 스타일을 맞춘다.
```
{
  "cue": "<짧은 한국어 맥락 큐 — 뜻을 한눈에 구분. 예: '위험으로부터','기계·엔진이','색이 연한'>",
  "ko": "<한국어 뜻, 사전식 간결체. 예: '달리다','아끼다, 절약하다'>",
  "pos": "<'동사'|'명사'|'형용사'|'부사'|'전치사'|'접속사' 등 한국어 품사명>",
  "lv": <정수 빈도레벨: 가장 흔한 뜻 1~3, 덜 흔할수록 크게>,
  "cefr": "<'A1'|'A2'|'B1'|'B2'|'C1'|'C2' — 이 '의미' 자체의 난이도. 표제어 전체의 CEFR과 다를 수 있음
           (예: run="달리다"는 A1급이지만 run="출마하다"는 B2~C1급). lv(빈도순서)와는 별개 축 —
           흔한 뜻이라고 무조건 쉬운 건 아님(예: bank="은행"은 lv1이지만 그 자체로 A1~A2 수준).
           일상 기초 어휘=A1~A2, 중급 일반 어휘=B1~B2, 전문·격식·추상 어휘=C1~C2 기준으로 판단.>",
  "en": "<그 의미의 학습자용 쉬운 영영 정의 한 문장>",
  "syn": ["<이 '의미'로 쓰였을 때 ex[0]의 문장에 그대로 바꿔 넣어도 자연스러운 단어만. WordNet
          wn의 해당 synset syn을 후보로 쓰되, 문맥에 안 맞으면 빼라 — 예: run(달리다)의 예문에
          jog는 자연스럽지만 run(운영하다) synset의 operate는 이 의미엔 안 맞으니 넣지 말 것.
          자연스러운 게 없으면 빈 배열 []. 3~5개 상한, 과분할 방지.>"],
  "ex": [{ "en": "<그 의미를 보여주는 자연스러운 예문>", "ko": "<자연스러운 한국어 번역>" }]
}
```
`syn`은 **그 의미 하나에 한정된** 문맥 치환어다(단어 전체의 유의어 목록이 아님) — 사전 상세 화면의
"유의어·반의어"(표제어 전체, WordNet 전 의미 취합)와는 다른 용도이니 혼동하지 말 것.

## 프로젝트 규칙 (CLAUDE.md)
- 한국어: 뜻·큐는 사전식 간결체(파일럿 참고).
- **이모지 절대 금지**(🎉⚠️ 등 어디에도).
- 예문은 해당 의미를 실제로 보여주는 자연스러운 문장, 한국어 번역도 자연스럽게(직역·딱딱한 투 금지).

## 출력 파일 (정확히 이 둘만 작성, 그 외 어떤 파일도 건드리지 말 것)
1. `/home/user/EN_vocab/pipeline/senses_parts/<BATCH>.json` — `{ "<소문자 word>": [ {sense}, ... ], ... }`. 배치의 **모든 단어** 포함(최소 1뜻).
2. `/home/user/EN_vocab/pipeline/review_parts/<BATCH>.json` — `{ "<소문자 word>": "<한국어 사유>", ... }`. **확신 낮은 단어만**(희귀·전문·의미경계 모호·한국어 뜻 불확실·WordNet이 비어 추측). 나중에 사람/사전 대조용. 없으면 `{}`.

## 엄수
- merge_cefrj.js 실행·words.js 재생성·커밋/푸시 **금지**. index.html 등 지정 두 파일 외 **어떤 것도 수정 금지**.
- 배치 단어 **100% 커버**. 하나도 건너뛰지 말 것.
- 유효한 JSON만(파싱 확인).
- **중간 저장 필수(세션 리밋 대비)**: 이 작업은 도중에 API 세션 리밋으로 강제 종료될 수 있다. 배치를
  끝까지 다 처리한 뒤 마지막에 한 번에 파일을 쓰면, 중간에 끊겼을 때 그동안의 작업이 통째로 유실된다.
  **25~30단어 단위로 끊어서 그때그때 senses_parts/<BATCH>.json·review_parts/<BATCH>.json을
  갱신(기존 내용에 누적, 유효한 JSON 유지)**해라. 언제 중단되어도 그 시점까지 결과물이 파일에 남도록.

## 반환 (최종 메시지, 간결히)
- 처리 단어 수 + 의미 개수 분포(예: "1뜻:40, 2뜻:55, 3뜻:30, 4+뜻:12").
- 검수 플래그 개수.
- 가장 흥미로운 다의어 분리 2개의 senses[] 전문(품질 스팟체크용).
