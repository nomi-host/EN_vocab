실제 배포 에셋은 `../../img/hero/` (cat.json / cat_run.json / smile.json / sticker_*.png / ground.png /
ghost_thumbnail.png / lottie.min.js). 여기는 재생성용 소스/스크립트.

## 방식(요약 — 자세한 건 auto-memory "최종 확정 방법")
- 캐릭터 = 이미지-에셋 Lottie + **하드컷**(크로스페이드 금지: 손그림 다리 잔상). 프레임에 그려진
  먼지 등은 **그대로** 사용(별도 먼지 레이어 X).
- 렌더러 = lottie-web(자체호스팅). 히어로 배경/땅/스티커는 앱 DOM으로 별도 구현(index.html HeroScene).

### 박스 재계산(2026-07-19, 2차 — "캐릭터가 훨씬 크고 땅으로 더 내려와있다" 지적)
- **레퍼런스 실측 방법**: 사용자 스크린샷 2장이 350:220(hero-scene) 비율의 타이트한 크롭인지 먼저
  검증(이미지 자체 종횡비 1.586 ≈ 350/220=1.591, 룰드라인 간격으로도 교차검증) → 맞다고 확인되면
  픽셀좌표를 곧바로 scene%로 환산 가능. 캐릭터 실루엣만 골라내려고 (검정 잉크|주황 발|흰 배) 색
  마스크 + `scipy.ndimage.label` 연결요소 분석으로 스티커·먼지·땅라인과 분리(땅라인이 먼지/캐릭터
  잉크와 겹치며 하나로 이어붙는 문제가 있어, 몸통은 땅라인 위쪽 구간에서만 찾고 발끝(주황)만 별도로
  전체 높이에서 찾는 2단계 방식 사용). 결과: 걷기 실루엣 bbox = scene 기준 (28.14, 26.76, 69.33,
  93.92)%, 달리기 = (28.27, 30.13, 69.8, 92.18)%.
- **왜 박스가 실루엣 %보다 훨씬 커야 하는가**: `.hero-cat`은 실루엣이 아니라 **comp 전체(200×200,
  내부에 캐릭터 잉크+여백 포함)**가 들어갈 자리 — 캐릭터 잉크는 comp의 일부만 차지(걷기 프레임1
  기준 세로 약 75%, `y_shift` 적용 후 comp 내 상대위치도 같이 이동). "실루엣 목표 %"를 그대로
  박스에 넣으면 여백까지 같이 줄어들어 캐릭터가 훨씬 작게 나옴(1차 시도에서 실제로 겪음 — 걷기
  기준 목표 대비 실측 결과가 사방으로 10%p 안쪽에 그침).
- **역산**: 각 세트의 "대표 포즈"(포즈1, `goToAndStop(0)`으로 프레임 고정 실측 — 유니온 bbox가 아니라
  단일 프레임 bbox를 씀, 유니온은 꼬리 등이 벌어진 프레임까지 합쳐져 레퍼런스의 정지 포즈와 종횡비가
  안 맞음) 잉크 bbox(다시 걷기: 프레임1 native `viewBox` 기준 x 40.3~186/y 24~174.7, 달리기: 프레임1
  기준 x 13.7~176.3/y 29.3~169.3, 단 `y_shift` 반영해 실제 comp 내 좌표로 환산)를 이용해 세로 기준
  스케일(`scale = 목표높이 / (잉크세로비율×comp세로)`)을 구하고, `박스 = comp세로(200×scale)`로
  정사각형 취급(가로 쏠림 오차는 감수 — 달리기는 다리를 더 벌린 포즈라 잉크 자체 종횡비가 레퍼런스
  정지 포즈와 안 맞아 가로가 약 3~4%p 더 넓게 나옴, 세로/접지 위치가 우선순위라 판단).
  최종값: **걷기** `.hero-cat{left:16.85%;top:11.2%;width:56%;height:88.8%}`(기본 CSS),
  **달리기** `HeroScene`에서 `stage>=2`일 때 인라인 스타일로
  `{left:20.16%;top:10.5%;width:55.7%;height:88.6%}` 오버라이드(최초 24.45%였는데 "우측으로
  쏠려 보인다"는 지적으로 15px=4.29%p 좌측 이동).
- 이전 "접지 8유닛 이동" 보정은 이번 박스 재계산에 흡수됨 — `build_cat_lottie.py`의 `y_shift`
  파라미터(걷기 11 / 달리기 15, comp 200 기준)가 그 역할을 이어받아 "캐릭터 잉크가 comp의 어느
  위치에 있는지"를 정의하고, 그 값 그대로 위 박스 역산에 반영됨. 코드를 다시 만질 땐 반드시
  **박스 값과 `y_shift`를 같이** 바꿔야 함(따로 바꾸면 접지·크기 둘 다 어긋남).

### 3단계(날기) — 2026-07-19 착수
- **소스**: `../rough/ghost-fly-01~10.svg`(10프레임, 전부 동일 `viewBox="0 0 200 200"`). **1~4번(쪼그렸다
  점프해 뜨는 인트로)은 1회만 재생, 5~10번(대각선으로 나는 포즈+파티클 잔상 6종)을 무한 루프.**
- **재생 구조 = Lottie 2개(인트로/루프 분리), 하나의 comp에 넣고 loop 세그먼트로 자르는 방식 안 씀**
  — `img/hero/cat_fly_intro.json`(4프레임, `loop:false`)을 먼저 재생 → `complete` 이벤트에서
  `img/hero/cat_fly_loop.json`(6프레임, `loop:true`)으로 교체 마운트. `LottieBox`는 항상
  `loop:true`라 재사용 불가해 전용 컴포넌트 `FlyCat`(index.html) 신설. 두 세트 다 포즈당 3프레임
  (FPS30) — 인트로 12프레임(0.4s, 1회), 루프 18프레임(0.6s/loop, 걷기·달리기와 같은 하드컷 방식,
  미세 bob 포함).
- **박스 계산도 같은 comp≠실루엣 역산 방법을 썼지만, 이 포즈는 대각선이라 레퍼런스 실측 종횡비
  (0.915, 세로가 더 김)와 각 프레임 자체 종횡비(루프 포즈는 전부 1.05~1.22, 가로가 더 김)가 크게
  어긋나 있었음** — 세로를 정확히 맞추려면 comp가 scene 세로의 100%를 넘고 박스 top이 음수(약
  -14%)가 돼야 해서(즉 카드 위로 넘침) 포기하고, **걷기·달리기와 같은 스케일(≈0.98)로 타협** —
  결과적으로 "레퍼런스 원본 실측값과 정확히 일치"보다 "이전 두 단계와 비슷한 크기감"을 우선한 것.
  최종값: `left:26.13%;top:1%;width:56%;height:89.1%`. 대표 프레임은 루프의 3번째 포즈(원본
  `ghost-fly-07`, native bbox x 12.7~198/y 41.3~192.7)로 캘리브레이션.
- **땅 3단 상태머신(정지→가속→정속)**: 인트로(점프) 동안은 땅이 완전히 멈춰 있다가, 루프 시작과
  동시에 `ease-in`으로 서서히 가속해(`heroGroundRamp` 키프레임, 0.6s, 임의 튜닝값) 목표 속도
  (**달리기의 3배** = `FLY_GROUND_DUR = RUN_GROUND_DUR/3` ≈ 1.556s/loop)에 도달, 이후 무한 정속
  루프로 이어붙음. React state machine(`flyGroundPhase`: stopped→ramp→steady)으로 구현 — `FlyCat`의
  `onIntroEnd` 콜백이 stopped→ramp를, ramp 키프레임의 `onAnimationEnd`가 ramp→steady를 트리거.
  **이음매 무점프 트릭**: ramp가 `-var(--rampDist)%`(20%, 임의값)에서 끝나도록 만들고, steady
  단계의 `heroGround` 키프레임(0→-50% 무한반복)에 `animation-delay: -(rampDist/50)*steadyDur`
  (음수)를 줘서 "이미 그 위상에서 재생 중이던 것"처럼 시작 — 두 애니메이션의 실제 translateX 값이
  전환 순간 정확히 일치해 눈에 띄는 끊김 없음(Playwright로 전후 스크린샷 비교해 확인).
- **버그 발견**: `ground.png` CSS 캐시무효화 태그(`img/hero/ground.png?v=...`)가 **plain CSS라
  `HERO_V` JS 상수와 별개로 하드코딩돼 있어서, 지난 세션의 땅선 굵기 수정(v0.35.44) 이후에도
  버전 번호가 안 올라가 있었음(캐시된 브라우저는 얇아진 선을 못 받았을 수 있음)** — 이번에 발견해
  v0.35.48로 동기화. **앞으로 버전 올릴 때 `index.html` 317번째 줄 근처 `ground.png?v=` 문자열도
  반드시 같이 검색해서 갱신할 것**(HERO_V 치환 스크립트에 이 라인도 포함시켜뒀음).
